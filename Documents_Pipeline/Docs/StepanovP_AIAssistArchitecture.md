# 1. Общая схема потока данных

```
┌─────────────────────────┐     ┌─────────────────────────┐     ┌─────────────────────────┐
│      Purgatory          │     │    Promotion Service    │     │    Knowledge Base       │
│      (Staging)          │     │      (Async Worker)     │     │      (nsi.* Schema)     │
│                         │───▶│                          │───▶│                         │
│ • Реестр документов     │     │ • Валидация JSON Schema │     │ • hybrid_search         │
│ • Метаданные & статусы  │     │ • Генерация embeddings  │     │ • chunks + embeddings   │
│ • chunk_containers.json │     │ • Атомарная запись в БД │     │ • document_sections     │
│ • Assets (S3/MinIO)     │     │ • Граф связей (pending) │     │ • images / tables       │
│ • Audit & Review log    │     │ • Retry & Idempotency   │     │ • chunk_relations       │

   → status = 'approved'        → {doc_id, checksum, version}     → Answer + Citations
```

## 2.1. Purgatory (Чистилище)

Буферный слой приёма и первичной обработки. Фиксирует уникальность документа, хранит все исходные представления (скан, цифра, CAD, LaTeX), запускает OCR/парсинг, выделяет структуру, таблицы, формулы и изображения. Формирует текстовые чанки по пунктам, строит иерархию, выявляет ссылки на другие НСИ, фиксирует метаданные (версии, поправки, актуальность, источник). Результат работы — строго типизированный chunk_containers.json + ссылки на ассеты в S3.

## 2.2. Promotion Service

Асинхронный воркер, который проверяет контейнер чанка на соответствие JSON-схеме, вычисляет векторные представления (embeddings) для каждого чанка и атомарно заносит данные в рабочую схему nsi.*: текст и векторы → chunks, метаданные → documents, иерархию → document_sections, изображения → images, таблицы → extracted_tables. Строит граф связей, обрабатывая «висячие» ссылки (когда целевой документ ещё не загружен) через механизм pending_relations и периодический резолвер.

## 2.3. Knowledge Base (База знаний)

Принимает запросы пользователя, выполняет гибридный поиск: векторный по embeddings, BM25 по полнотекстовому индексу tsv в PostgreSQL, нечёткий поиск через pg_trgm. Контекст расширяется по графу связей на заданную глубину. Найденные фрагменты ранжируются (RRF-фьюжн), объединяются в структурированный пакет и передаются в LLM. Формируется ответ с обязательными ссылками на конкретные документы, пункты и страницы.

# 3. Детальная проработка архитектуры

## 3.1. Purgatory (Детализация)

| Задача | Техническая реализация |
|---|---|
| Уникальность | SHA-256 файла + нормализация метаданных. При совпадении хэша → линк к существующей записи, дедупликация на уровне реестра. |
| OCR & Парсинг | Docling / Marker для layout-анализа, PaddleOCR для сканов, TableTransformer для таблиц. Извлечение сразу в Markdown/JSON. |
| Иерархия | Парсинг оглавления + эвристика нумерации (1., 1.1, а) → формирование ltree-путей в structure блока. |
| Ассеты (S3) | Изображения, чертежи, формулы хранятся в S3/MinIO. В JSON остаются только s3_key, mime, checksum, bbox и привязки. |
| Ссылки на НСИ | RegEx/NER: /(ГОСТ\|ОСТ\|РД\|СП)\s[\d\.\-]+/ → извлечение target_doc_code + target_section. Сохраняются как pending_references. |
| Версионность | Поля valid_from, valid_to, superseded_by, amendments[]. Авто-алерты при приближении срока замены. |
| Выход | Строго валидируемый chunk_containers.json + файлы в s3://purgatory/{doc_id}/{version_sha256}/... |

## 3.2. Promotion Service (Механизмы)

| Механизм | Описание |
|---|---|
| Валидация | fastjsonschema + кастомные правила: последовательность chunk_index, пороги confidence, корректность ltree-путей, связи в relations. |
| Эмбеддинги | Генерируются внутри воркера перед транзакцией (model.encode). В JSON не хранятся. При падении модели → retry. |
| Атомарная запись | Всё (documents → sections → chunks → images → tables) в одной PostgreSQL-транзакции. Сбой → ROLLBACK, статус в Purgatory → failed. |
| Граф связей | from_chunk → to_chunk. Если целевой документ отсутствует → запись в nsi.pending_relations (status='pending'). |
| Resolver (Cron) | Celery Beat (~4ч): ищет совпадения pending_relations.target_doc_code, создаёт chunk_relations/document_relations, меняет статус на resolved. |
| Идемпотентность | Проверка purgatory_source_id в nsi.documents.meta. Повторный вызов задачи не создаёт дубликаты. |

## 3.3. Knowledge Base (Поиск и RAG)

| Компонент | Реализация |
|---|---|
| Где BM25? | В PostgreSQL: nsi.chunks.tsv (tsvector) + GIN-индекс. S3 не индексирует текст. |
| Нечёткий поиск | pg_trgm + оператор % (word_similarity). Бонусный коэффициент в RRF. |
| Гибридный поиск | Vector (HNSW) + BM25 (tsvector) + Fuzzy (trgm) → RRF Fusion (k=60). |
| Расширение контекста | Рекурсивный CTE по chunk_relations (глубина 1–2, score > 0.6). Метка related_via: reference/conflict. |
| Пакет для LLM | JSON: {"instruction", "context": [{"doc","section","text","page","relation"}], "query"}. LLM только форматирует ответ. |
| Кэширование | nsi.query_cache (хеш запроса + чексумма источников). TTL 7 дней. Инвалидация при обновлении чанков. |

# 4. Стратегия хранения файлов

| Хранилище | Роль | Почему |
|---|---|---|
| S3 / MinIO | Исходные PDF/DOCX/CAD, извлечённые ассеты, превью | Отказоустойчиво, масштабируемо, версионирование, pre-signed URLs. |
| PostgreSQL | Метаданные, chunk_containers.json, статусы, tsv, embeddings | ACID, гибридный поиск, граф, атомарность, аудит. |
| НЕ используется | BLOB в БД, локальный диск сервера | Раздувают БД, ломают бэкапы, нет дедупликации, единая точка отказа. |

## Паттерн именования в S3:

s3://nsi/{env}/{doc_uuid}/v{version_sha256}/{asset_type}/{file_hash}.{ext}

*Оригинальные имена файлов НЕ используются в путях. Они хранятся только в метаданных БД. Это исключает конфликты имён и гарантирует контент-адресуемость.*
