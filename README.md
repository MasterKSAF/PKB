# OKB RAG Frontend

Frontend для локальной RAG-системы инженерного конструкторского бюро. Интерфейс сделан под требования ТЗ: ввод запроса, диалог, ответ со страницами и документами, история запросов, реестр документов, сверка параметров и QA-контур.

## Что есть в UI

- `Чат`: основной диалог в стиле ChatGPT, статусы ответа, ссылки `Страница` и `Документ`, просмотр источника в отдельной панели.
- `Поиск`: поиск по базе знаний с фрагментами и релевантностью.
- `Реестр`: список документов, OCR-статусы, индексация и действия администратора базы знаний.
- `Сверка`: сравнение параметров по источникам и статусы расхождений.
- `История`: запрос, ответ, пользователь, источники, статус, подготовка под поиск и экспорт.
- `QA`: контрольные метрики, оценка ответов и журнал проверки.
- `Видеоинструкция`: полноэкранный demo-walkthrough для показа сценария работы.

## Локальный запуск

```bash
cd frontend
npm install
npm run dev
```

По умолчанию приложение открывается на:

```text
http://localhost:3000
```

Проверка:

```bash
cd frontend
npm run build
npm run lint
```

## Переменные окружения

Создать `frontend/.env.local`:

```env
VITE_API_BASE_URL=http://localhost:8000
```

Если переменная не задана, frontend использует `http://localhost:8000`.

## Docker

Сборка образа:

```bash
cd frontend
docker build -t okb-rag-frontend .
```

Запуск:

```bash
docker run --rm -p 3000:80 okb-rag-frontend
```

## Что нужно от backend

Backend должен быть локальным или размещенным в контуре заказчика. Ожидаемый стек: `FastAPI`, `PostgreSQL`, `pgvector`, `Celery + Redis`, OCR/RAG pipeline.

Ключевые требования:

- все ответы API возвращаются в JSON;
- ошибки возвращаются в едином формате;
- каждый RAG-ответ должен содержать документ, страницу, раздел и фрагмент;
- если данных недостаточно, backend возвращает `needs_clarification`;
- если источники конфликтуют, backend возвращает `source_conflict`;
- для точного поиска нужен hybrid retrieval: keyword search, vector search, metadata filters и rerank.

## Основные API

- `POST /chat`
- `GET /search`
- `GET /documents`
- `GET /source-preview`
- `GET /checks`
- `GET /history`
- `POST /feedback`
- `GET /metrics`

## Демо-сценарий

1. Инженер открывает `Чат` и задает вопрос.
2. Ассистент при необходимости просит уточнить контекст.
3. После уточнения система возвращает короткий ответ и ссылки на страницы и документы.
4. Инженер открывает страницу или документ и сверяет фрагмент.
5. Тимлид смотрит `Историю`, `Реестр` и `QA`, чтобы оценить качество работы системы.
