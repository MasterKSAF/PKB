# PKB RAG Frontend

Frontend для локальной RAG-системы по инженерным документам и нормативно-справочной информации. Интерфейс собран как демонстрационный рабочий контур под ТЗ: чат с источниками, поиск, реестр документов, проверка на соответствие требованиям НСИ, история запросов и QA-блок.

## Что есть в UI

| Раздел | Что делает |
|---|---|
| `Чат` | Основной сценарий работы инженера. Ответы показываются вместе со ссылками `Страница` и `Документ`, а сами страницы и документы открываются в правой панели предпросмотра. |
| `Поиск` | Поиск по базе знаний с фрагментами, релевантностью и открытием страницы или полного документа. |
| `Реестр` | Список документов базы знаний, OCR-статусы, индекс, действия по загрузке и обновлению. |
| `Проверка` | Сопоставление проектного решения с требованиями НСИ. Есть выбор документов, пакетная проверка, открытие страниц и документов, выгрузка результата в Excel. |
| `История` | Журнал запросов, ответов, пользователей и статусов с фильтрами и экспортом. |
| `QA` | Контрольные метрики, оценка ответов ассистента и журнал проверки. |
| `Фокус-режим` | Доступен на каждой вкладке и скрывает левую панель, чтобы оставить только текущую рабочую область. |
| `Видеоинструкция` | Полноэкранный demo-сценарий интерфейса. |

## Режим работы

UI может работать без backend в mock/demo-режиме. Это позволяет открыть интерфейс локально, пройтись по вкладкам, проверить сценарии и кнопки, а также показать продукт команде до готовности API.

При подключении backend интерфейс использует те же экраны и ту же структуру пользовательского сценария, но уже с реальными данными.

## Docker-запуск

Из папки `frontend`:

```bash
docker compose up --build
```

После запуска открыть:

```text
http://localhost:3000
```

Остановить контейнер:

```bash
docker compose down
```

## Что нужно от backend

Backend должен работать локально или в контуре заказчика. Базовая ожидаемая схема: API для чата, поиска, реестра документов, проверки, истории и метрик.

| Что нужно | Зачем это нужно в UI |
|---|---|
| JSON-ответы по всем API | Чтобы frontend стабильно отображал данные без ручной обработки форматов. |
| Привязка ответа к документу, странице, разделу и фрагменту | Чтобы инженер видел, на чем основан ответ ассистента. |
| Стабильные идентификаторы или URL страницы и документа | Чтобы ссылки `Страница` и `Документ` открывали правильное превью. |
| Признак необходимости уточнения | Чтобы чат не угадывал недостающий контекст, а просил инженера уточнить проект, раздел или документ. |
| Статус конфликта источников | Чтобы UI показывал ситуацию, когда найденные документы противоречат друг другу. |
| История запросов и ответов | Чтобы вкладка `История` показывала реальные обращения, источники, пользователей и статусы. |
| Контрольные метрики и журнал | Чтобы вкладка `QA` показывала качество поиска, OCR, ответов и фидбэка инженеров. |

Для вкладки `Проверка` backend должен отдавать результат сопоставления проектного решения с требованиями НСИ.

| Поле | Смысл |
|---|---|
| `project` | Проект или документ, который проверяется. |
| `section` | Раздел проектной документации. |
| `parameter` | Проверяемый параметр. |
| `projectValue` | Значение, найденное в проекте. |
| `nsiRequirement` | Требование НСИ, с которым идет сверка. |
| `nsiDocument` | Документ НСИ, где найдено требование. |
| `status` | Результат проверки: `OK`, `WARNING` или `ERROR`. |
| `comment` | Пояснение для инженера. |

## Основные API

| Метод | Endpoint | Для чего используется |
|---|---|---|
| `POST` | `/chat` | Отправка вопроса инженера и получение ответа ассистента. |
| `GET` | `/search` | Поиск по базе знаний. |
| `GET` | `/documents` | Получение реестра документов. |
| `GET` | `/source-preview` | Получение превью страницы или полного документа, если backend отдает превью отдельным endpoint. |
| `GET` | `/checks` | Получение результатов проверки НСИ. |
| `GET` | `/history` | Получение истории запросов и ответов. |
| `POST` | `/feedback` | Отправка оценки ответа инженером. |
| `GET` | `/metrics` | Получение QA-метрик и журнальных записей. |

## Демонстрационный сценарий

1. Инженер открывает `Чат` и задает вопрос по документам или нормам.
2. Система возвращает ответ и ссылки `Страница` и `Документ`.
3. Инженер открывает нужную страницу или полный документ в правой панели.
4. При необходимости инженер переходит в `Поиск`, `Проверку` или `Реестр`.
5. Тимлид или аналитик смотрит `Историю` и `QA`, чтобы оценить качество работы системы.
