# OKB RAG Frontend

Frontend для локальной RAG-системы по инженерным документам и нормативно-справочной информации. Интерфейс собран как демонстрационный рабочий контур под ТЗ: чат с источниками, поиск, реестр документов, проверка на соответствие требованиям НСИ, история запросов и QA-блок.

## Что есть в UI

- `Чат`: основной сценарий работы инженера. Ответы показываются вместе со ссылками `Страница` и `Документ`, а сами страницы и документы открываются в правой панели предпросмотра.
- `Поиск`: поиск по базе знаний с фрагментами, релевантностью и открытием страницы или полного документа.
- `Реестр`: список документов базы знаний, OCR-статусы, индекс, действия по загрузке и обновлению.
- `Проверка`: сопоставление проектного решения с требованиями НСИ. Есть выбор документов, пакетная проверка, открытие страниц и документов, выгрузка результата в Excel.
- `История`: журнал запросов, ответов, пользователей и статусов с фильтрами и экспортом.
- `QA`: контрольные метрики, оценка ответов и журнал проверки.
- `Фокус-режим`: доступен на каждой вкладке и скрывает левую панель, чтобы оставить только текущую рабочую область.
- `Видеоинструкция`: полноэкранный demo-сценарий интерфейса.

## Режим работы

UI может работать без backend в mock/demo-режиме. Это позволяет:

- открыть интерфейс локально;
- пройтись по вкладкам;
- проверять сценарии и кнопки;
- показывать продукт команде до готовности API.

При подключении backend интерфейс должен использовать те же экраны и ту же структуру пользовательского сценария, но уже с реальными данными.

## Локальный запуск

```bash
cd frontend
npm install
npm run dev
```

По умолчанию приложение доступно на:

```text
http://localhost:3000
```

Проверка сборки и типов:

```bash
cd frontend
npm run build
npm run lint
```

## Переменные окружения

Создать `frontend/.env.local`:

```env
VITE_API_BASE_URL=http://localhost:8000
```

Если переменная не задана, frontend использует `http://localhost:8000`.

## Docker

Сборка образа:

```bash
cd frontend
docker build -t okb-rag-frontend .
```

Запуск:

```bash
docker run --rm -p 3000:80 okb-rag-frontend
```

Если порт `3000` занят:

```bash
docker run --rm -p 3001:80 okb-rag-frontend
```

## Что нужно от backend

Backend должен работать локально или в контуре заказчика. Базовая ожидаемая схема: API для чата, поиска, реестра документов, проверки, истории и метрик.

Для корректной интеграции нужны следующие вещи:

- все ответы API приходят в JSON;
- у каждого ответа ассистента есть привязка к документу, странице, разделу и текстовому фрагменту;
- для страницы и документа есть стабильные идентификаторы или URL предпросмотра;
- если данных недостаточно, backend возвращает признак необходимости уточнения;
- если источники конфликтуют, backend возвращает отдельный статус конфликта;
- для вкладки `Проверка` backend отдает результат сопоставления:
  - проект;
  - раздел;
  - параметр;
  - значение в проекте;
  - требование НСИ;
  - документ НСИ;
  - статус `OK / WARNING / ERROR`;
  - комментарий;
- для `Истории` backend хранит запрос, ответ, пользователя, статус и источники;
- для `QA` backend отдает контрольные метрики и журнальные записи.

## Основные API

- `POST /chat`
- `GET /search`
- `GET /documents`
- `GET /source-preview`
- `GET /checks`
- `GET /history`
- `POST /feedback`
- `GET /metrics`

## Демонстрационный сценарий

1. Инженер открывает `Чат` и задает вопрос по документам или нормам.
2. Система возвращает ответ и ссылки `Страница` и `Документ`.
3. Инженер открывает нужную страницу или полный документ в правой панели.
4. При необходимости инженер переходит в `Поиск`, `Проверку` или `Реестр`.
5. Тимлид или аналитик смотрит `Историю` и `QA`, чтобы оценить качество работы системы.
