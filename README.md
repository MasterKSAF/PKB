# PKB RAG Frontend

Frontend для локальной RAG-системы по инженерным документам и нормативно-справочной информации. Интерфейс собран как демонстрационный рабочий контур под ТЗ: чат с источниками, поиск по базе знаний, реестр документов, проверка проектных решений на соответствие требованиям НСИ, история запросов, QA-метрики и администрирование.

Основная текущая версия находится в папке **[frontend](frontend)** и собрана на React + Vite + MUI.

## Что есть в UI

| Раздел | Что делает |
|---|---|
| `Чат` | Основной сценарий работы инженера: вопрос, ответ ассистента, нумерация выводов, ссылки `Страница` и `Документ`, открытие материалов в правой панели предпросмотра. |
| `Поиск` | Поиск по базе знаний: запрос, фильтры, результаты с релевантностью, фрагментами и открытием страницы или полного документа. |
| `Реестр` | Список документов базы знаний: статусы OCR, индексация, загрузка документов, обновление индекса и переобработка OCR. |
| `Проверка` | Сверка проектного решения с требованиями НСИ: выбор одного или нескольких документов, пакетная проверка, открытие проектных и нормативных источников, выгрузка результата в Excel. |
| `История` | Журнал запросов и ответов: пользователь, статус, источники, фильтры и экспорт текущей выборки. |
| `QA` | Контроль качества: метрики OCR, поиска, ответов со страницами, оценка ответов ассистента инженерами и журнал проверки. |
| `Администрирование` | Управление пользователями, ролями, правами доступа, административным журналом и настройками доступа к документам, OCR-артефактам и журналам. |
| `Фокус-режим` | Скрывает левую навигацию и оставляет только текущую рабочую область. Доступен на каждой вкладке. |
| `Видеоинструкция` | Полноэкранный demo-сценарий интерфейса. |

## Роли и доступ

В интерфейсе есть переключение пользователя в правой верхней части рабочей области. У пользователя отображаются ФИО, должность и роль. Доступные вкладки зависят от выбранной роли.

| Роль | Доступные вкладки |
|---|---|
| `Инженер` | `Чат`, `Поиск`, `Проверка`, `История`. |
| `Администратор знаний` | `Чат`, `Поиск`, `Реестр`, `Проверка`, `История`, `QA`, `Администрирование`. |
| `Администратор системы` | `Чат`, `Поиск`, `Реестр`, `Проверка`, `История`, `QA`, `Администрирование`. |

В текущей frontend-версии роли работают в demo-режиме через mock-данные. После подключения backend роль должна приходить из профиля пользователя или auth-сервиса.

## Темы и режимы

Интерфейс поддерживает две темы:

| Режим | Назначение |
|---|---|
| `Темная тема` | Основной визуальный режим для демонстрации и повседневной работы. |
| `Светлая тема` | Альтернативный режим для светлого рабочего окружения. |

Переключение темы находится в верхней панели рабочей области. Фокус-режим работает независимо от выбранной темы.

## Режим работы

UI может работать без backend в mock/demo-режиме. Это позволяет:

- открыть интерфейс локально;
- пройтись по вкладкам;
- проверить сценарии и кнопки;
- показать продукт команде до готовности API;
- обсудить UX с заказчиком без развернутого backend-контура.

При подключении backend интерфейс использует те же экраны и пользовательские сценарии, но уже с реальными данными.

## Docker-запуск

Из папки `frontend`:

```bash
docker compose up --build
```

После запуска открыть:

```text
http://localhost:3000
```

Остановить контейнер:

```bash
docker compose down
```

## Что нужно от backend

Backend должен работать локально или в контуре заказчика и отдавать данные в JSON. UI ожидает API для чата, поиска, документов, проверки, истории, обратной связи, QA-метрик и администрирования.

| Что нужно | Зачем это нужно в UI |
|---|---|
| Профиль пользователя: ФИО, должность, роль | Чтобы показывать пользователя в верхней панели и включать нужные вкладки по роли. |
| Список ролей и прав доступа | Чтобы вкладка `Администрирование` управляла доступом к документам, OCR-артефактам, журналам и системным настройкам. |
| JSON-ответы по всем API | Чтобы frontend стабильно отображал данные без ручной обработки форматов. |
| Ответ ассистента с привязкой к документу, странице, разделу и фрагменту | Чтобы инженер видел, на чем основан ответ. |
| Стабильные ID или URL страницы и документа | Чтобы кнопки `Страница` и `Документ` открывали корректное превью справа. |
| Признак необходимости уточнения | Чтобы чат мог попросить инженера уточнить проект, раздел или документ, если данных недостаточно. |
| Статус конфликта источников | Чтобы UI показывал ситуацию, когда найденные документы противоречат друг другу. |
| История запросов, ответов, пользователей и источников | Чтобы вкладка `История` показывала реальные обращения и позволяла фильтровать/экспортировать журнал. |
| QA-метрики и журнальные записи | Чтобы вкладка `QA` показывала качество OCR, поиска, ответов и инженерских оценок. |
| Административный журнал | Чтобы администратор видел изменения пользователей, ролей, прав и настроек. |

Для вкладки `Проверка` backend должен отдавать результат сопоставления проектного решения с требованиями НСИ.

| Поле | Смысл |
|---|---|
| `project` | Проект или документ, который проверяется. |
| `section` | Раздел проектной документации. |
| `parameter` | Проверяемый параметр. |
| `projectValue` | Значение, найденное в проекте. |
| `nsiRequirement` | Требование НСИ, с которым идет сверка. |
| `nsiDocument` | Документ НСИ, где найдено требование. |
| `status` | Результат проверки: `OK`, `WARNING` или `ERROR`. |
| `comment` | Пояснение для инженера. |

## Основные API

| Метод | Endpoint | Для чего используется |
|---|---|---|
| `POST` | `/chat` | Отправка вопроса инженера и получение ответа ассистента. |
| `GET` | `/search` | Поиск по базе знаний. |
| `GET` | `/documents` | Получение реестра документов. |
| `GET` | `/source-preview` | Получение превью страницы или полного документа. |
| `GET` | `/checks` | Получение результатов проверки НСИ. |
| `GET` | `/history` | Получение истории запросов и ответов. |
| `POST` | `/feedback` | Отправка оценки ответа инженером. |
| `GET` | `/metrics` | Получение QA-метрик и журнальных записей. |
| `GET` | `/users` | Получение пользователей для администрирования. |
| `GET` | `/roles` | Получение ролей и прав доступа. |
| `POST` | `/admin/access` | Сохранение изменений прав доступа. |
| `GET` | `/admin/audit-log` | Получение административного журнала. |

Названия endpoint пока рабочие. При интеграции с реальным backend их можно заменить в `frontend/src/utils/http.ts`.

## Демонстрационный сценарий

1. Инженер открывает `Чат` и задает вопрос по документам или нормам.
2. Ассистент возвращает нумерованный ответ и ссылки `Страница` / `Документ`.
3. Инженер открывает нужную страницу или полный документ в правой панели.
4. При необходимости инженер переходит в `Поиск` или `Проверку`.
5. Администратор знаний смотрит `Реестр`, `QA` и `Администрирование`.
6. Тимлид или аналитик смотрит `Историю` и `QA`, чтобы оценить качество работы системы.
