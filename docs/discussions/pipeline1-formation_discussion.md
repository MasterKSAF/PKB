По итогам встречи была согласована архитектура, в которой обработка документа разбивается на четыре основных микросервиса (плюс оркестратор). Ниже — описание функционала каждого сервиса на основе расшифровки.

---

## 1. Сервисы распознавания и извлечения (сырых данных)

**Кто выполняет:**  
1.OCR для сканов / изображений.
2.Парсер для цифровых PDF и DOC.

**Вход:**  
Исходный файл (скан, PDF, DOC).

**Выход:**  
«Сырой» JSON — упрощённая структура, состоящая из массива страниц и блоков на них.

**Содержимое блока:**

- тип (текст, таблица, фигура, формула, картинка)
- текст/содержимое (если есть)
- координаты (bbox) в нормированных координатах (0..1, верхняя левая точка, Y вниз)
- ссылка на оригинал (FileKey) — если это изображение или таблица, требующая сохранения
- другие простые признаки, которые можно извлечь без сложного анализа

**Ответственность:**

- не строит иерархию, не выделяет заголовки/абзацы, не связывает таблицы, разорванные на страницах
- не использует LLM
- сохраняет бинарные объекты (картинки, таблицы как картинки) в файловое хранилище и возвращает FileKey
- не знает о других документах (не проверяет ссылки, не определяет ГОСТ)

---

## 2. Сервис конвертации / валидации (логический парсер)

**Кто выполняет:**  
«Конвертер-валидатор» (название рабочее; внутри может быть и логический парсер, и валидатор, при необходимости разделят).

**Вход:**  
Сырой JSON от сервиса №1 (может приходить как от OCR, так и от парсера PDF).

**Выход:**  
Иерархический, типизированный JSON — уже близкий к итоговому документу-контейнеру.

**Что делает:**

- преобразует плоский список блоков в иерархическую структуру (разделы, подразделы, абзацы, заголовки)
- объединяет таблицы, разорванные на нескольких страницах (понимает по признакам, что это одна таблица)
- выделяет метаданные (обозначение документа, название, даты)
- распознаёт перекрёстные ссылки на другие документы (например, ГОСТ …) и проставляет их
- может использовать LLM для:
  - построения иерархии
  - распознавания типов документов
  - нормализации структуры
- выполняет валидацию (проверка полноты, соответствия схеме)
- при необходимости проверяет уникальность документа (через отдельный вызов своего же API для «первичных метаданных»)
- сохраняет промежуточные результаты в журнале оркестратора

**Важно:**  
Этот сервис **не** сохраняет данные в итоговое векторное хранилище — он готовит структурированное представление.

## 3. Сервис реестра документов (Registry)

**Кто выполняет:**  
Сервис, отвечающий за запись в основную базу данных и подготовку данных для индексации.

**Вход:**  
Выходной JSON от сервиса №2 (уже иерархический, с метаданными и ссылками).

**Выход:**  
«Плоский» JSON, максимально приближенный к схеме БД, содержащий только то, что нужно для:

- хранения карточки документа
- построения векторных чанков (для поиска)

**Что делает:**

- формирует карточку документа (ID, обозначение, название, срок действия и т.д.)
- сохраняет документ в реестре (если он новый)
- разбивает документ на секции (чанки) с привязкой к ID документа
- **не включает** в выходной JSON информацию, необходимую только для отображения (полные картинки, графику и т.п.) — только текст и ссылки на FileKey, если они нужны для поиска
- сохраняет связи с другими документами (cross‑references), если они нужны для индексации

---

## 4. Оркестратор (Pipeline)

**Задача:**  
Управляет последовательностью вызовов сервисов и хранит историю обработки каждого документа.

**Особенности из обсуждения:**

- может дважды вызывать один и тот же сервис (например, сначала получить «первичные метаданные» из сервиса №1 и №2 для проверки уникальности, а потом запустить полную обработку)
- ведёт журнал (лог) всех этапов, который не входит в реестр документов, но доступен для анализа
- обеспечивает передачу FileKey между сервисами, а не самих бинарных данных

---

## Дополнительные сквозные решения

### Bbox (координаты)

- Нормированные координаты (0..1), верхняя левая точка (0,0), ось Y вниз.
- Передаются на всех этапах, но на этапе индексации могут быть отброшены.
- Подсветка на UI — опционально (кнопка), пока не реализуется в первую очередь.

### FileKey (работа с файловым хранилищем)

- Везде вместо путей к файлам используется `FileKey` (хэш / GUID).
- Общий модуль работы с хранилищем (каждый сервис его подключает) ответственен за генерацию уникальных ключей, типы файлов и пр.

### Карточка документа и превью

- «Зародыш» карточки (превью) формируется в начале пайплайна на основе первых страниц.
- Полная карточка появляется после сервиса №2 (валидатора) и окончательно сохраняется в реестре.
- Для превью используются те же сервисы, но с отдельным API («дай первичные метаданные», «проверь уникальность»).

### Уникальность документа и версионность

- Решение о бизнес-ключе (как считать документы одинаковыми, учёт редакций, префиксы ГОСТ/ГОСТ‑Р/ГОСТ‑ИСО) **отложено**.
- Механизм для проверки заложен, но точные правила будут доработаны позже (возможно, через LLM).

### Protected spans

- Пока оставлено как абстрактное пожелание (запрет разрывать определённые диапазоны текста/таблиц при чанковании).
- Реализация не обязательна для первого этапа.

## Итоговая структура микросервисов (принципиальная схема)

```text
[Исходный файл]
     │
     ▼
┌─────────────────────────┐
│ Сервисы №1              │
│ OCR и Parser (PDF)      │
│ → сырой JSON (блоки)    │
└─────────────────────────┘
     │
     ▼
┌─────────────────────────┐
│ Сервис №2               │
│ Конвертер + валидатор   │
│ → иерархический JSON    │
│ (использует LLM)        │
└─────────────────────────┘
     │
     ▼
┌─────────────────────────┐
│ Сервис №3               │
│ Registry / реестр       │
│ → плоский JSON для БД   │
│ и векторных чанков      │
└─────────────────────────┘
     │
     ▼
[База данных + векторное хранилище]
```

Все сервисы взаимодействуют через оркестратор, который управляет пайплайном и журналирует события.
