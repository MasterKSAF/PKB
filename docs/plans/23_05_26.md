## 1. Ключевые архитектурные расхождения

### 1.1 Двухфазный пайплайн (preview + full)

**Эталон** (`pipeline1-formation_detail.md`, раздел 4):
- Фаза превью: OCR/Parser (preview-режим, первые N страниц) → Converter-validator (preview API: метаданные + проверка уникальности) → **решение пользователя** (продолжить/стоп/новая версия)
- Фаза полной обработки: OCR/Parser (полный режим) → Converter-validator (полный режим: иерархия, LLM, валидация) → Registry → RAG Builder

**Текущая документация** (все файлы, кроме detail):
- Линейный single-pass: Parsing → Validation → Registry
- Нет preview-фазы
- Нет точки принятия решения пользователем между фазами
- `review_required` — единственный механизм вмешательства, и он внутри валидации, а не между фазами

**Затронутые файлы**: `pipeline1-formation.md`, `overview.md`, `orchestrator_service_api.md`, `common.md`, все FSM-диаграммы

---

### 1.2 Состав сервисов

**Эталон** (раздел 3):
| Сервис | Назначение |
|--------|------------|
| OCR-сервис | Сканы/изображения |
| Parser-сервис | Цифровые PDF, DOC, DOCX |
| Converter-validator | Конвертация + валидация (единый сервис) |
| Registry | Реестр документов, сегментация |
| RAG Builder | Чанкование, эмбеддинги, векторный индекс |
| RAG Search | Поиск по индексу |

**Текущая документация**:
| Сервис | Назначение |
|--------|------------|
| Parsing / OCR Service | Распознавание + парсинг (единый) |
| Validation Service | Только валидация |
| Registry Service | Реестр |
| RAG Service | Индексация + поиск (единый) |

**Расхождения**:
1. **OCR и Parser слиты в один сервис** — эталон требует два независимых с единым контрактом выходных данных
2. **Validation отделён от конвертации** — эталон объединяет в Converter-validator
3. **RAG — монолитный** — эталон разделяет Builder и Search

**Затронутые файлы**: `pipeline1-formation.md`, `pipeline2-indexation.md`, `pipeline3-search.md`, `overview.md`, `ocr_service_api.md`, `validate_service_api.md`, `rag_service_api.md`, `common.md`

### 1.3 Зона ответственности: сегментация vs чанкование

**Эталон** (разделы 3.3, 3.4):
- **Registry**: «Сегментирование: разбивает документ на логические секции-сегменты с привязкой к ID документа»
- **RAG Builder**: «Чанкование: разбивает документ на логические секции-чанки с привязкой к ID сегментов и документа»

**Проблема**: в эталоне формулировки пересекаются — оба говорят «разбивает на секции». Уточнено устно (см. раздел 5.1): Registry → секции (DB-запись `nsi_document_sections`), RAG Builder → более мелкие чанки для эмбеддингов.

**Текущая документация**:
- `pipeline1-formation.md`: Registry «сохраняет секции», чанкование не упоминается
- `pipeline2-indexation.md`: RAG Indexing «построение чанков», секции не упоминаются
- `pipeline1-formation_detail.md`: оба упоминают разбиение — формулировки нужно уточнить

**Затронутые файлы**: `pipeline1-formation_detail.md` (уточнить формулировки), `pipeline2-indexation.md`, `overview.md`

---

### 1.4 Preview-данные: где хранятся

## 2. Расхождения API-контрактов

### 2.1 OCR/Parser: нет preview-эндпоинтов

**Эталон** (раздел 3.1):
- Preview-режим: первые N страниц, без сохранения бинарных объектов, без `fileKey`
- Полный режим: все страницы, бинарные объекты → `fileKey`

**Текущий API** (`ocr_service_api.md`):
- `POST /ocr/process` — только полный режим
- Нет параметра `max_pages`
- Нет разделения на OCR и Parser

**Необходимо**:
- Добавить `POST /ocr/preview` и `POST /parser/preview`
- Или добавить параметр `mode: "full" | "preview"` + `max_pages`
- Создать отдельный `parser_service_api.md`

---

### 2.2 Converter-validator: нет preview API и нет endpoint'а конвертации

**Эталон** (раздел 3.2):
- Preview API: `извлечение первичных метаданных` + `быстрая проверка уникальности`
- Полный режим: построение иерархии, LLM, валидация

**Текущий API** (`validate_service_api.md`):
- `POST /validate/document` — валидация (не конвертация)
- `POST /validate/classifiers`
- `POST /validate/check`
- `POST /validate/extract/parameters`
- Нет endpoint'а построения иерархии
- Нет preview-эндпоинтов
- Нет параметров LLM

**Необходимо**:
- `POST /converter/preview/metadata`
- `POST /converter/convert` (полная конвертация)
- Оставить существующие `/validate/*` в составе полного цикла

---

### 2.3 RAG: монолитный сервис вместо двух

**Эталон** (разделы 3.4, 3.5):
- RAG Builder: строит индекс
- RAG Search: ищет по индексу

**Текущий API** (`rag_service_api.md`):
- Один сервис с режимами A (Indexing) и B (Search)
- Одни и те же endpoint'ы `/rag/index` и `/rag/search`

**Необходимо**:
- Разделить на `rag_builder_service_api.md` и `rag_search_service_api.md`
- Разные порты

---

### 2.4 Оркестратор: нет preview-эндпоинтов

**Текущий API** (`orchestrator_service_api.md`):
- `POST /documents` — загрузка и линейный запуск
- `GET /documents/{doc_id}/status` — статус без preview-шагов
- Нет эндпоинта для решения пользователя

**Необходимо**:
- `POST /documents/{doc_id}/preview`
- `GET /documents/{doc_id}/preview/status`
- `POST /documents/{doc_id}/decide` (action: proceed / stop_duplicate / force_new_version)
- В `GET .../status` добавить блок `preview` в `steps.pipeline.formation`

## 3. Расхождения форматов данных

### 3.1 Координаты bbox: пиксели vs нормированные 0..1

**Эталон** (раздел 2 «Bbox»):
> Нормированные координаты (0..1), начало в левом верхнем углу (0,0), ось Y направлена вниз.

**Эталонные JSON'ы** (`schema_parser_result.json`, `schema_converter_result.json`, `schema_registry_for_rag.json`):
- Все используют **пиксельные** координаты: `"bbox": [10, 10, 100, 18]`

**Статус**: ❓ **ОТКРЫТЫЙ ВОПРОС** — нуждается в решении:
- Вариант A: привести JSON'ы к 0..1
- Вариант B: допустить пиксели на сыром этапе, нормировка в Converter-validator
- Вариант C: оставить как есть, добавить примечание

---

### 3.2 Именование поля для бинарных ссылок: `image_key` vs `file_key`

**Эталон** (раздел 2 «FileKey»):
> Все пути к файлам заменены на `fileKey`

**Эталонные JSON'ы**:
- `schema_parser_result.json`: использует `"image_key": "purgatory/assets/.../eq1.png"`
- `schema_converter_result.json`: микс — где-то `image_key`, где-то `file_key`
- `schema_registry_for_rag.json`: `"image_key": "purgatory/assets/.../t1.png"` и `"file_key"` в разных местах

**Необходимо**: унифицировать на `file_key` во всех JSON'ах.

**В `/schema` нет файла** с примером такого ответа. Рекомендуется создать `schema_parser_preview.json`.

## 5. Устные уточнения (зафиксированы 23.05.26)

### 5.1 Сегментация vs чанкование

- **Registry** делит документ на **секции** (= записи в `nsi_document_sections`)
- **RAG Builder** бьёт секции на более мелкие **чанки** (для эмбеддингов)
- Термины «сегментирование» и «чанкование» обозначают разные операции, в эталоне формулировки нужно уточнить

### 5.2 OCR и Parser — два независимых сервиса

- Разные endpoint'ы
- Единый контракт выходных данных (сырой JSON)
- Аналогично RAG Builder и RAG Search — два разных сервиса

### 5.3 Preview-данные

- Хранятся в Оркестраторе как блок данных в журнале пайплайна
- Отдельная таблица БД не требуется
- У каждого пайплайна свой журнал операций

### 5.4 Порты

- Назначаются последовательно, начиная с 8081 (Orchestrator)
- Исходные порты сохранены; новые сервисы (Parser, RAG Builder, RAG Search) добавлены по возрастанию
- Фактическая схема (актуальная):

| Сервис | Порт | Примечание |
|--------|------|------------|
| Orchestrator | 8081 | Исходный |
| Auth | 8082 | Исходный |
| Query Service | 8083 | Исходный |
| Registry | 8084 | Исходный |
