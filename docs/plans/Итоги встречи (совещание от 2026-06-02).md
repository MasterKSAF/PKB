## Итоги встречи (совещание от 2026-06-02)

### 1. Основные решения

#### 1.1. Работа с репозиторием и ветками Git

- **Проблема**: Игорь залил свой код (RagBuilder) в личную папку (`Абзалов Игорь/Source/RagBuilder`), а не в общий каталог `backend`.

- **Решение**:
  
  - Все сервисы должны находиться в каталоге `backend`. Личные папки в корне репозитория недопустимы.
  
  - Игорь скопирует папку `RagBuilder` в `backend` и закоммитит.
  
  - **Правило ветвления**: изменения начинаются с **малой ветки** (личной) → затем в `develop/сервис` → затем в общий `develop`. Нельзя делать много папок в одной ветке.
  
  - Ветки делаются личными, папки – общими. Это предотвращает конфликты.

#### 1.2. Обработка редакций и замен документов

- **Контекст**: циркулярные письма, частичные изменения, полные замены документов (ГОСТы, морской регистр).

- **Принятое решение** (ключевое):
  
  - Ввести **диапазон дат действия** документа (начало – конец).
  
  - При появлении нового документа (редакции или замены) создаётся **новая копия** документа с новым диапазоном дат. У старого документа проставляется дата окончания действия.
  
  - Поиск будет учитывать диапазон дат (фильтрация по дате проверки).
  
  - **Терминология** (договорились не путать):
    
    - Черновики (ранее Версия) (промежуточные результаты распознавания).
    
    - *Редакция* – частичное изменение существующего документа.
    
    - *Новый документ* – полная замена предшествующего.
    
    - *Полная перезапись* – ручная замена документа из-за ошибки загрузки (не связана с изменениями в законодательстве).

#### 1.3. Схема обработки документов

- Загрузка файла → регистрация → хранилище черновиков (несколько вариантов распознавания) → выбор финального «чистовика» (человеком или LLM) → определение типа (новый документ / редакция / замена) → помещение в реестр.

- Полная перезапись возможна только по решению человека (автомат не сможет определить).

#### 1.4. Хранение данных

- **[Min.IO](https://min.io/)** – для файлов (исходники PDF, картинки, черновики). Негуманно для людей, но хорошо для программ.

- **Реляционная база** – для реестра документов (ссылки на ключи [Min.IO](https://min.io/), метаданные, диапазоны дат). Большие объекты (BLOB) в базу не класть.

- **GitHub** – для кода, документации, обсуждений. Ссылки на облачные диски (Яндекс.Диск, Google) удобнее для людей, чем прямые ссылки на [Min.IO](https://min.io/).

#### 1.5. Распознавание и артефакты

- Разные варианты одного документа (сканы, цифровые слои с артефактами, OCR‑ошибки). Предложено:
  
  - Использовать **LLM** для выбора наилучшего варианта (сравнение текста, таблиц, цифр).
  
  - В админке предусмотреть ручной выбор и возможность правки (например, исправление «ГОСГ» → «ГОСТ»).

- Термин «артефакт» признан неоднозначным (побочный эффект vs. историческая ценность). Рекомендовано использовать **«результат обработки»**.

#### 1.6. Структура данных

- Исходный формат – **JSON** (хранит максимум деталей). Из него генерируется MD, HTML и т.д.

- Файловая структура (предложенная Павлом) удобна для человека, но не для машинной обработки. Хранить её в базе не планируется, только в [Min.IO](https://min.io/) как черновики.

### 3. Особенности / проблемы

- **Плохая связь** у Игоря (прерывания звука, пропадание). Договорились выходить на связь позже при улучшении условий.

- **Отсутствие участников**: Виталий не присоединился (поиск), Дмитрий Кептилов упоминается как экспериментирующий с LLama Parser.

- **Вопрос с редакциями** остаётся открытым в части автоматического определения замены по циркулярным письмам. Пока решили не париться на MVP – обрабатывать такие письма как обычные документы, а в будущем доработать механизм.

- **Неоднозначность терминов** (артефакт, версия, редакция) – договорились придерживаться выработанной терминологии, чтобы не путать новых участников.

### 4. Дополнительные технические замечания

- В [Min.IO](https://min.io/) используются «бакеты» (корзины) и ключи доступа. Версионирование файлов поддерживается.

- При полной перезаписи документа ссылки на него сохраняются (лучше, чем удаление + создание).

- Распознанные таблицы, строки, JSON-поля – всё это будет храниться в БД в виде структурированных данных, а не файловой размазни.
