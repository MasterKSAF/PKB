## 2. Ключевые архитектурные решения (зафиксированы)

### 2.1. Идентификаторы
- `task_id`, `session_id`, `message_id` — **bigint** (строго возрастающие).
- `document_id`, `version_id` — **bigint**.
- `comparison_id`, `batch_id`, `export_id`, `feedback_id`, `file_key` — **требуют определения**.

### 2.2. Версионность документов
- **Черновики** (drafts) — промежуточные результаты распознавания (привязаны к `task_id`).
- **Редакции** (editions) — частичные изменения существующего документа (диапазон дат действия).
- **Новый документ** — полная замена предшествующего.
- **Полная перезапись** — ручная замена из-за ошибки загрузки.
- Механизм редакций **отложен для MVP**, но заложен в архитектуру.

### 2.3. Пайплайн обработки документа (Pipeline 1)
```
[Файл] → OCR/Parser → Converter-Validator → Registry → [БД + MinIO]
```
- Двухфазный: **preview** (первичные метаданные) → **full** (полная обработка).
- Оркестратор управляет последовательностью, журналирует этапы.
- Сервисы взаимодействуют через **Long Polling**.
- `FileKey` (хеш/GUID) — единый механизм ссылок на файлы в MinIO.

### 2.4. Пайплайн индексации (Pipeline 2)
- Оркестратор вызывает RagBuilder сразу после записи в Registry.
- **Scheduler — внутри оркестратора** (не отдельный сервис).
- Таймаут `pending_index` — 48 часов.

### 2.5. Query-сервис
- Полный pipeline обработки чата внутри себя (включая LLM).
- **Микропайплайн**: запрос → история чата → извлечение → обработка.
- 4 метода API для точечных обновлений чата.
- Гибридный поиск: векторный + полнотекстовый BM25 + цитирование.
- **Привязка к проектам**: сессии чата (`chat.sessions`) содержат `project_id` → `chat.projects.id`, что позволяет группировать чаты по судостроительным проектам и фильтровать историю.

## 4. Инструкции по реализации (для разработчиков)

### 4.1. Правила работы с репозиторием

1. **Структура**: все сервисы — в `backend/<service_name>/`. Личные папки в корне недопустимы.
2. **Ветвление**: личная ветка → `develop/<service>` → общий `develop`.
3. **Коммиты** — в ветку `develop` (или feature-ветку) до понедельника.
4. **Деплой**: коммит в `release-<service>` триггерит авто-деплой через GitHub Actions.
5. **Образец `pipeline.yaml`** — взять у Игоря, адаптировать под свой сервис.

### 4.2. Требования к каждому сервису

Обязательные артефакты в репозитории сервиса:

```
backend/<service_name>/
├── Dockerfile              # Сборка контейнера
├── pipeline.yaml           # CI/CD (на основе образца)
├── .env.example            # Пример конфига с описанием переменных
├── src/                    # Исходный код
├── tests/                  # Юнит-тесты (запуск через pytest)
├── docs/                   # Описание API и логики (секвенс-диаграммы, пайплайны)
└── README.md               # Как запустить локально, как использовать
```

### 4.3. Конфигурация сервиса

Каждый сервис должен поддерживать переключение режимов через `.env`:

```env
# Режим работы (заглушка / реальный сервис)
USE_MOCK_RAG_BUILDER=false
USE_MOCK_RAG_SEARCH=false
USE_MOCK_REGISTRY=false

# Режим эмбеддингов
EMBEDDING_MODE=local         # local | server
EMBEDDING_MODEL=Qwen/Qwen2.5-4B-Instruct
EMBEDDING_SERVER_URL=http://infinity:7997

# База данных
DATABASE_URL=postgresql://user:pass@localhost:5432/pkb

# MinIO
MINIO_ENDPOINT=localhost:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=minioadmin
```

### 4.4. Контракты API

- Формат обмена: JSON.
- Все ID: `task_id`, `session_id`, `message_id` — **bigint** (не строки).
- `document_id`, `version_id` — **bigint**.
- Координаты bbox: **нормированные [0,1]** на всех этапах. Конвертация пикселей в нормализованные координаты выполняется внутри OCR/Parser.
- Ссылки на файлы: только через `FileKey` (хеш/GUID).
- Единая схема выхода OCR/Parser: `raw_ocr_v4`.
- Вложенная структура JSON для OCR и Parser обязательна.

### 4.5. Документирование сервиса

По образцу документации Игоря (отмечена как эталонная):
- Чёткое API с сигнатурами запросов/ответов.
- Секвенс-диаграммы взаимодействия (Mermaid).
- Описание пайплайна внутри сервиса.
- Коды ошибок и их значения.
- **Важно**: качественная документация защищает от деградации кода при использовании AI-агентов (vibe coding).

### 4.6. Логирование

- Использовать единую библиотеку: **`loguru`** (рекомендация).
- Уровни: DEBUG (разработка), INFO (продакшен).
- Все сервисы отправляют логи в централизованную систему (OpenTelemetry → Grafana).
- Диана предоставляет middleware для подключения.

### 4.7. Тестирование

- Юнит-тесты для каждого сервиса — **обязательны**.
- Запуск: `pytest` (локально, до Docker).
- Интеграционные тесты — через `ServiceChecker` (расширенная версия).
- Полный сценарий: загрузка → парсинг → индексация → поиск.

### 4.8. Работа с эмбеддингами

- Протокол: **OpenAI-совместимый API** (совместимость с Ollama, Qwen, Infinity и внешними API).
- Локальная модель: **Qwen 4B** (или 8B при наличии GPU).
- Продакшен-сервер: **Infinity** (Linux, Docker с персистентным хранилищем).
- На Windows — тестировать через локальную загрузку модели в Python.
- Качество > дешевизна: маленькие бесплатные модели не обеспечивают нормального поиска.

### 4.9. База данных

- **PostgreSQL 15/16 + pgvector**.
- Схемы: `purgatory` (черновики) и `public` (основная БД).
- Каждый сервис имеет доступ только к своим схемам. Чужие данные — только через API.
- Миграции: пока **одноразовые** (пересоздание БД). После фиксации структуры → патчевые.
- BLOB-объекты в БД не хранить — использовать MinIO + `FileKey`.
