### 🟢 A2. bbox: пиксели в raw_ocr_v4, нормированные в validated_v3 (решено)

- `schema_parser_result.json` (raw_ocr_v4): bbox в пикселях — `[10.0, 10.0, 100.0, 18.0]`, страница 210×297 px (`page.width`, `page.height`).
- `schema_converter_result.json` (validated_v3): bbox нормированные — `[0.05, 0.028, 0.5, 0.05]`.
- `glossary.md` (04.06, исправлено): bbox описан для обоих форматов.

**Решение (уточнено 04.06)**: bbox в OCR/Parser — сырые пиксели (px), в Converter — нормированные [0,1]. Конвертация пикселей в нормализованные координаты выполняется внутри Converter-validator. Единицы измерения в API исправлены с "мм" на "пиксели (сырые, px)" для OCR/Parser, и на "нормализованные (0..1)" для Converter.
**Статус**: ✅ Закрыто.

### 🟢 A3. Версия raw_ocr: план говорит v2, документация — v4 (решено)

**Статус**: ✅ Закрыто.

**Статус**: закрыто 04.06.

### 🟡 A5. Черновики (drafts) и purgatory schema не задокументированы (пробел)

- **Проблема**: сводный план (разделы 2.2, 4.9) и встреча 02.06 описывают механизм черновиков: несколько вариантов распознавания, привязка к `task_id`, выбор финального «чистовика». Однако ни в `pipeline1-formation.md`, ни в `glossary.md`, ни в API-файлах эта механика не была раскрыта.

**Решение**: создан план [`docs/plans/drafts_storage_plan.md`](../plans/drafts_storage_plan.md) (уточнён 04.06):
  - Черновики — часть Оркестратора (схема `pipeline`, таблица `pipeline.drafts`), НЕ отдельное приложение
  - `file_key` создаётся при загрузке в MinIO, передаётся в task + draft
  - Черновик-пайплайн = preview-фаза; после него — промотирование в Registry
  - Если документ уникален и чист → **автопромот**; при аномалиях → **решение человека**
  - История хранится, лимита нет, человек может удалить вручную
  - `file_key` — у черновика; `task.data` — для промежуточных JSON
  - Глубина обработки: OCR/Parser (full) в черновике, Converter-validator при промотировании
  - Базовое описание добавлено в `pipeline1-formation.md` и `glossary.md`
  - **05.06:** API-эндпоинты черновиков документированы в `orchestrator_service_api.md` (группа `drafts`, 5 эндпоинтов), FSM черновиков визуализирован в `pipeline1-formation.md`, маршрут `/api/v1/drafts/*` добавлен в Gateway

**Статус**: 🟢 документировано. Осталось: UI сравнения черновиков (Спринт 2), решение по аномалиям (до 10.06).

### 🟢 A7. Отсутствие сущности «проект» в чатах (решено)

- **Проблема**: Gateway и QueryService не поддерживали сущность «проект» (судостроительный заказ). Чаты не сгруппированы по проектам, фильтры истории не позволяют отобрать чаты по проекту.
- **Решение**: введена таблица `chat.projects` (bigint id, code, name, status) и поле `project_id` (bigint FK) в `chat.sessions`. API QueryService обновлён: `project_id` в создании/редактировании сессии, фильтр `project_id` в списке сессий и истории.
- **Статус**: ✅ Закрыто. Термин "Проект" добавлен в `glossary.md`, схема БД обновлена (`db_diagrams.md`). Реализация в плане Спринта 1 (04.06–10.06).
