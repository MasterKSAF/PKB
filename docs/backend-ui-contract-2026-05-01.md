# Backend-контракт для полной работоспособности UI

Документ описывает, что нужно от backend, чтобы frontend `PKB RAG Frontend` работал не в demo/mock-режиме, а как полноценный интерфейс локальной RAG-системы по инженерным документам и НСИ.

Основная идея: UI не должен сам принимать инженерные решения, хранить нормативную базу или выполнять поиск. UI показывает рабочие экраны, отправляет действия пользователя в backend и отображает результат с привязкой к документам, страницам, ролям, истории и QA-метрикам.

## 1. Общие требования к backend

| Требование | Что нужно реализовать |
|---|---|
| Единый API | REST API по базовому адресу из `VITE_API_BASE_URL`, сейчас по умолчанию `http://localhost:8000`. |
| Формат данных | Все основные ответы в JSON, документы и экспортные файлы отдельными binary/blob-ответами. |
| Локальный контур | Backend должен разворачиваться локально или на сервере заказчика без внешних облачных зависимостей. |
| Стабильные ID | У пользователя, документа, страницы, чанка, ответа, источника, проверки и feedback должен быть стабильный идентификатор. |
| Роли и доступы | Backend должен отдавать роль пользователя и доступные вкладки/действия. UI не должен быть единственным местом проверки прав. |
| Источники | Каждый ответ ассистента и каждая проверка должны иметь ссылки на документ, страницу, раздел и фрагмент текста. |
| Превью документов | Backend должен уметь отдавать отдельную страницу и полный документ для правой панели предпросмотра. |
| История | Backend должен сохранять запрос, ответ, пользователя, источники, статус, время, feedback и аналитику. |
| Ошибки | Ошибки должны приходить в едином формате, чтобы UI мог показать понятное сообщение пользователю. |
| Производительность | Поиск и открытие превью должны работать быстро: с пагинацией, кэшированием и потоковой отдачей больших документов. |

## 2. Базовая модель ошибок

Для всех endpoint желательно использовать единый формат:

```json
{
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "Документ не найден",
    "details": {
      "documentId": "doc-123"
    },
    "requestId": "req-20260501-001"
  }
}
```

Рекомендуемые HTTP-статусы:

| Статус | Когда использовать |
|---|---|
| `400` | Некорректный запрос или не хватает обязательных параметров. |
| `401` | Пользователь не авторизован. |
| `403` | У пользователя нет прав на действие или вкладку. |
| `404` | Документ, страница, проверка или пользователь не найдены. |
| `409` | Конфликт версий, прав или источников. |
| `422` | Документ принят, но не может быть обработан из-за формата или структуры. |
| `500` | Внутренняя ошибка backend. |

## 4. Глобальные функции UI

| Функция UI | Что делает пользователь | Что нужно от backend |
|---|---|---|
| Статус системы | UI показывает online/offline/demo. | `GET /health` должен возвращать состояние API, БД, индекса, OCR-очереди и хранилища документов. |
| Выбор пользователя/роли | В demo UI можно переключать пользователя сверху. | В боевом режиме роль должна приходить из `GET /auth/me` или аналогичного endpoint. Ручное переключение можно оставить только для demo/admin. |
| Доступность вкладок | UI скрывает вкладки по роли. | Backend должен отдавать список permissions и дополнительно проверять права на каждом endpoint. |
| Темная/светлая тема | Пользователь переключает тему. | Backend не обязателен. Можно хранить локально в браузере. Если нужно синхронизировать между устройствами, нужен `PATCH /users/{id}/preferences`. |
| Фокус-режим | Скрывает левую панель. | Backend не нужен. Это локальное состояние UI. |
| Видеоинструкция | Открывает полноэкранное видео. | Backend не нужен, если файл лежит как статический asset. Если видео хранится на сервере, нужен `GET /media/ui-demo`. |

## 5. Вкладка `Чат`

Главная задача вкладки: инженер задает вопрос, получает нумерованный ответ, видит ссылки `Страница` и `Документ`, открывает их справа и оставляет оценку ответа.

### Кнопка/действие: отправить вопрос

| Параметр | Описание |
|---|---|
| UI-действие | Пользователь вводит вопрос и нажимает отправку. |
| Endpoint | `POST /chat` или потоковый вариант `POST /chat/stream`. |
| Backend делает | Выполняет поиск по базе знаний, формирует ответ, связывает каждый пункт ответа с источником. |
| UI ожидает | Текст ответа, статус, timestamp, список источников, связь источников с пунктами ответа. |

Пример запроса:

```json
{
  "q": "Какая минимальная толщина листа требуется для корпуса проекта 21900?",
  "conversationId": "conv-001",
  "projectId": "project-21900",
  "documentIds": [],
  "userId": "u1",
  "options": {
    "topK": 5,
    "requireCitations": true,
    "stream": false
  }
}
```

Пример ответа:

```json
{
  "id": "msg-102",
  "role": "assistant",
  "status": "answered",
  "content": "1. Минимальная толщина листа не должна определяться отдельно от проекта.\n2. Для несущих элементов требуется сверка с разделом корпуса.\n3. Если редакции документов расходятся, решение передается на инженерную проверку.",
  "timestamp": "13:02",
  "answerItems": [
    {
      "index": 1,

      "citationIds": ["cit-001"]

  "citations": [
    {
      "id": "cit-001",
      "documentId": "doc-001",
      "document": "Правила классификации и постройки морских судов",
      "version": "2026",
      "section": "Часть II. Корпус",

    "retrievalMs": 840,
```

### Статусы ответа

| Status | Что показывает UI |
|---|---|
| `answered` | Ответ найден, есть источники. |
| `needs_clarification` | Не хватает контекста: проект, раздел, документ, версия НСИ. |
| `insufficient_data` | В базе нет достаточных данных для ответа. |
| `source_conflict` | Найдены противоречивые источники или версии документа. |

### Кнопка `Страница`

| Параметр | Описание |
|---|---|
| UI-действие | Пользователь нажимает `Страница` под конкретным пунктом ответа. |
| Endpoint | `GET /documents/{documentId}/pages/{page}/preview` или `GET /source-preview?citationId=...&kind=page`. |
| Backend делает | Возвращает только нужную страницу с подсветкой/текстом фрагмента. |
| UI ожидает | Данные для правой панели: номер страницы, название документа, фрагмент, изображение/PDF-страница или HTML-preview. |

Пример ответа:

```json
{
  "kind": "page",
  "documentId": "doc-001",
  "page": 45,
  "title": "Правила классификации и постройки морских судов",
  "section": "Часть II. Корпус",
  "mimeType": "application/pdf",
  "pageImageUrl": "/documents/doc-001/pages/45/image",
  "text": "Распознанный текст страницы...",
  "highlight": {
    "fragment": "Фрагмент требования...",
    "boxes": []
  }
}
```

### Кнопка `Документ`

| Параметр | Описание |
|---|---|
| UI-действие | Пользователь нажимает `Документ`. |
| Endpoint | `GET /documents/{documentId}/file` или `GET /source-preview?citationId=...&kind=document`. |
| Backend делает | Возвращает полный документ с первой страницы, с прокруткой. |
| UI ожидает | PDF/DOCX/XLSX/DWG preview или безопасный URL/stream для встроенного просмотра. |

Требования:

- полный документ должен открываться с первой страницы;
- при прокрутке название документа должно оставаться вверху панели;
- для больших файлов нужна поддержка range-запросов или постраничной загрузки;
- если формат нельзя встроить напрямую, backend должен отдать конвертированное preview в PDF/изображения.

### Блок feedback `Полезен ли был ответ`

| Параметр | Описание |
|---|---|
| UI-действие | Инженер нажимает полезно/не полезно, вводит комментарий и отправляет. |
| Endpoint | `POST /feedback`. |
| Backend делает | Сохраняет оценку, связывает ее с ответом, пользователем, источниками и QA-метриками. |
| UI ожидает | Подтверждение сохранения и, при необходимости, обновление QA-метрик. |

Пример запроса:

```json
{
  "messageId": "msg-102",
  "conversationId": "conv-001",
  "userId": "u1",
  "useful": true,
  "comment": "Ответ точный, источник подходит.",
  "openedCitationIds": ["cit-001"],
  "createdAt": "2026-05-01T13:05:00+03:00"
}
```

Пример ответа:

```json
{
  "ok": true,
  "feedbackId": "fb-001",
  "affectsMetrics": ["usefulRate", "ratedAnswers"]
}
```

## 6. Вкладка `Поиск`

Задача вкладки: найти документы или фрагменты в базе знаний, показать релевантность, краткий фрагмент и открыть документ в правой панели.

### Кнопка `Найти`

| Параметр | Описание |
|---|---|
| Endpoint | `GET /search`. |
| Query params | `q`, `type`, `version`, `limit`, `offset`, опционально `projectId`, `source`, `dateFrom`, `dateTo`. |
| Backend делает | Ищет по текстовому индексу и/или векторному индексу, возвращает релевантные фрагменты. |
| UI ожидает | Список результатов с документом, фрагментом, релевантностью и ссылками на preview. |

Пример запроса:

```http
GET /search?q=сталь%20корпуса&type=PDF&version=actual&limit=20&offset=0
```

Пример ответа:

```json
{
  "items": [
    {
      "id": "result-001",
      "documentId": "doc-001",
      "name": "Правила классификации и постройки морских судов",
      "type": "PDF",
      "source": "РКО",
      "version": "2026",
      "section": "Часть II. Корпус",
      "page": 45,
      "fragment": "Фрагмент найденного текста...",
      "relevance": 0.92,
      "status": "verified"
    }
  ],
  "total": 1,
  "queryId": "search-001"
}
```

### Кнопка `Сбросить`

Backend не нужен. UI очищает строку поиска и фильтры локально.

### Кнопка `Расширенные фильтры`

Если фильтры будут функциональными, backend должен отдать справочники:

| Endpoint | Что возвращает |
|---|---|
| `GET /search/filters` | Типы документов, источники, проекты, версии, статусы, диапазоны дат. |

### Кнопка `Открыть документ`

Использует те же preview-endpoint, что и чат:

- `GET /documents/{documentId}/pages/{page}/preview`;
- `GET /documents/{documentId}/file`.

## 7. Вкладка `Реестр`

Задача вкладки: показать состав базы документов, статусы OCR, индексации, обновления и обработки.

### Загрузка списка документов

| Параметр | Описание |
|---|---|
| Endpoint | `GET /documents`. |
| Backend делает | Возвращает документы из БД метаданных. |
| UI ожидает | Таблицу документов и данные для верхних счетчиков. |

Пример ответа:

```json
{
  "items": [
    {
      "id": "doc-001",
      "name": "Правила классификации и постройки морских судов",
      "type": "PDF",
      "source": "РКО",
      "version": "2026",
      "ocrStatus": "completed",
      "indexStatus": "indexed",
      "updatedAt": "2026-04-30T12:20:00+03:00"
    }
  ],
  "summary": {
    "total": 1,
    "ocrCompleted": 1,
    "indexed": 1,
    "needAttention": 0
  }
}
```

### Кнопка `Загрузить документ`

| Параметр | Описание |
|---|---|
| Endpoint | `POST /documents`. |
| Body | `multipart/form-data`: файл, источник, версия, тип, проект, комментарий. |
| Backend делает | Сохраняет оригинал в файловом хранилище, создает запись в БД, ставит задачу OCR/парсинга/индексации. |
| UI ожидает | ID документа, начальный статус обработки и сообщение об успешной загрузке. |

Пример ответа:

```json
{
  "documentId": "doc-101",
  "status": "uploaded",
  "ocrStatus": "queued",
  "indexStatus": "not_indexed",
  "jobId": "job-ocr-101"
}
```

### Кнопка `Обновить индекс`

| Параметр | Описание |
|---|---|
| Endpoint | `POST /documents/reindex`. |
| Backend делает | Пересобирает поисковый/векторный индекс по выбранным или всем документам. |
| UI ожидает | `jobId`, статус очереди, затем обновленный статус индексации. |

Пример запроса:

```json
{
  "documentIds": ["doc-001", "doc-002"],
  "mode": "incremental"
}
```

### Кнопка `Переобработать OCR`

| Параметр | Описание |
|---|---|
| Endpoint | `POST /documents/ocr/reprocess`. |
| Backend делает | Повторно запускает OCR/структурный парсинг для выбранных документов. |
| UI ожидает | `jobId`, статус обработки и обновление таблицы после завершения. |

### Меню действий в строке документа

Если меню будет активным, нужны действия:

| Действие | Endpoint |
|---|---|
| Открыть документ | `GET /documents/{documentId}/file`. |
| Открыть метаданные | `GET /documents/{documentId}`. |
| Скачать оригинал | `GET /documents/{documentId}/download`. |
| Переиндексировать один документ | `POST /documents/{documentId}/reindex`. |
| Удалить или архивировать | `DELETE /documents/{documentId}` или `PATCH /documents/{documentId}` со статусом `archived`. |

## 8. Вкладка `Проверка`

Задача вкладки: выбрать один или несколько проектных документов, сопоставить проектное решение с требованиями НСИ, показать результат и дать выгрузку в Excel.

### Выбор документов для проверки

| Параметр | Описание |
|---|---|
| Endpoint | `GET /documents?scope=project`. |
| Backend делает | Возвращает документы, доступные для проверки. |
| UI ожидает | Список документов для autocomplete. |

### Кнопка `Загрузить документ`

| Параметр | Описание |
|---|---|
| Endpoint | `POST /checks/uploads` или общий `POST /documents` с типом `project`. |
| Backend делает | Загружает проектный документ, создает временный или постоянный documentId, запускает OCR/парсинг при необходимости. |
| UI ожидает | ID загруженного документа и возможность выбрать его для проверки. |

### Кнопка `Проверить выбранные`

| Параметр | Описание |
|---|---|
| Endpoint | `POST /checks`. |
| Backend делает | Извлекает параметры из проектных документов, ищет релевантные требования НСИ, сопоставляет значения, возвращает статус. |
| UI ожидает | Таблицу результатов: проект, раздел, параметр, значение проекта, требование НСИ, документ НСИ, статус, комментарий. |

Пример запроса:

```json
{
  "documentIds": ["doc-project-001", "doc-project-002"],
  "nsiScope": "active",
  "mode": "batch",
  "options": {
    "includeWarnings": true,
    "returnSources": true
  }
}
```

Пример ответа:

```json
{
  "checkRunId": "check-20260501-001",
  "status": "completed",
  "items": [
    {
      "id": "check-row-001",
      "projectDocumentId": "doc-project-001",
      "projectDocument": "Проект 21900. Раздел корпуса",
      "projectSection": "Корпус / листовая сталь",
      "projectPage": 12,
      "projectText": "Фрагмент проектного решения...",
      "parameter": "Толщина листа",
      "projectValue": "8 мм",
      "nsiRequirement": "Не менее 8 мм",
      "nsiDocumentId": "doc-nsi-001",
      "nsiDocument": "Правила классификации и постройки морских судов",
      "nsiSection": "Часть II. Корпус",
      "nsiPage": 45,
      "nsiText": "Фрагмент требования НСИ...",
      "status": "OK",
      "comment": "Проектное значение соответствует требованию НСИ."
    }
  ],
  "summary": {
    "ok": 1,
    "warning": 0,
    "error": 0
  }
}
```

### Статусы проверки

| Status | Значение |
|---|---|
| `OK` | Проектное решение соответствует требованию НСИ. |
| `WARNING` | Требуется инженерная проверка: не хватает данных, неоднозначный параметр, разные редакции. |
| `ERROR` | Найдено явное расхождение с требованием НСИ. |

### Ссылки `Страница` и `Документ` в таблице

Для каждой строки нужны ссылки отдельно на проектный документ и на документ НСИ:

| Ссылка | Что открывает |
|---|---|
| `Страница` у проекта | Страницу проектного документа, где найден параметр. |
| `Документ` у проекта | Полный проектный документ с первой страницы. |
| `Страница` у НСИ | Страницу нормативного документа с требованием. |
| `Документ` у НСИ | Полный документ НСИ с первой страницы. |

### Кнопка `Выгрузить в Excel`

Варианты реализации:

| Вариант | Описание |
|---|---|
| Клиентский экспорт | UI выгружает текущую таблицу, если все данные уже загружены. Сейчас так работает demo. |
| Backend-экспорт | `GET /checks/{checkRunId}/export?format=xlsx` возвращает готовый XLSX с форматированием, фильтрами и служебными данными. Предпочтительно для боевого режима. |

## 9. Вкладка `История`

Задача вкладки: хранить и показывать все запросы, ответы, пользователей, статусы, источники, feedback и экспортировать отфильтрованную выборку.

### Загрузка истории

| Параметр | Описание |
|---|---|
| Endpoint | `GET /history`. |
| Query params | `q`, `userId`, `status`, `dateFrom`, `dateTo`, `limit`, `offset`. |
| Backend делает | Возвращает историю с учетом фильтров и прав пользователя. |
| UI ожидает | Таблицу истории и общее количество записей. |

Пример ответа:

```json
{
  "items": [
    {
      "id": "hist-001",
      "createdAt": "2026-05-01T13:05:00+03:00",
      "userId": "u1",
      "user": "Иванов С.П.",
      "query": "Какая минимальная толщина листа?",
      "answer": "Краткий текст ответа...",
      "sources": 2,
      "status": "answered",
      "feedback": {
        "useful": true,
        "comment": "Ответ точный"
      }
    }
  ],
  "total": 1
}
```

### Фильтры

| Фильтр | Что делает backend |
|---|---|
| Поиск по запросу/ответу | Фильтрует `query` и `answer`. |
| Пользователь | Фильтрует по `userId`. |
| Статус | Фильтрует по `answered`, `needs_clarification`, `insufficient_data`, `source_conflict`. |

### Кнопка `Экспорт`

| Параметр | Описание |
|---|---|
| Endpoint | `GET /history/export`. |
| Query params | Те же фильтры, что в таблице. |
| Backend делает | Возвращает CSV/XLSX текущей выборки. |
| UI ожидает | Файл для скачивания. |

## 10. Вкладка `QA`

Задача вкладки: показать контрольные метрики качества, оценку ответов ассистента инженерами и журнал проверки.

### Загрузка метрик

| Параметр | Описание |
|---|---|
| Endpoint | `GET /metrics`. |
| Backend делает | Агрегирует метрики OCR, поиска, ответов, feedback и журнал событий. |
| UI ожидает | Числа для карточек, подписи, статусы и журнал. |

Пример ответа:

```json
{
  "controlMetrics": {
    "ocrQuality": 98.4,
    "retrievalQuality": 91.2,
    "answersWithSources": 100,
    "searchLatency": 1.4
  },
  "engineerRatings": {
    "usefulRate": 87.5,
    "ratedAnswers": 48,
    "flaggedForReview": 3,
    "unresolvedAfterReview": 1
  },
  "logs": [
    {
      "time": "12:34:02",
      "type": "retrieval",
      "text": "Поиск по запросу завершен. Подобрано 5 документов.",
      "level": "info"
    }
  ]
}
```

### Подсказки `i`

Подсказки можно оставить на frontend как статическое описание. Если backend-команда хочет управлять методиками расчета, нужен endpoint:

| Endpoint | Что возвращает |
|---|---|
| `GET /metrics/definitions` | Описание каждой метрики, формулу, целевое значение, источник данных. |

### Журнал проверки

| Endpoint | Что возвращает |
|---|---|
| `GET /metrics/logs?limit=50` | События поиска, OCR, переиндексации, feedback, ручной проверки, конфликтов источников. |

## 11. Вкладка `Администрирование`

Задача вкладки: управлять пользователями, ролями, правами доступа и смотреть административный журнал.

### Загрузка пользователей

| Параметр | Описание |
|---|---|
| Endpoint | `GET /admin/users`. |
| Backend делает | Возвращает пользователей, роли, должности, логины, статусы, последний вход. |
| UI ожидает | Таблицу пользователей и текущего выбранного пользователя. |

Пример ответа:

```json
{
  "items": [
    {
      "id": "u1",
      "name": "Иванов Сергей Петрович",
      "position": "Ведущий инженер-конструктор",
      "login": "ivanov.sp",
      "role": "Инженер",
      "status": "Активен",
      "lastSeen": "2026-05-01 11:40",
      "access": ["chat", "search", "checks", "history"]
    }
  ]
}
```

### Изменение роли

| Параметр | Описание |
|---|---|
| UI-действие | Администратор выбирает пользователя и меняет роль. |
| Endpoint | `PATCH /admin/users/{userId}`. |
| Backend делает | Сохраняет роль, пересчитывает права, пишет запись в audit-log. |
| UI ожидает | Обновленного пользователя и новую запись журнала. |

Пример запроса:

```json
{
  "role": "knowledge_admin"
}
```

### Чекбоксы прав доступа

| Параметр | Описание |
|---|---|
| UI-действие | Администратор включает/выключает доступ к разделам. |
| Endpoint | `PUT /admin/users/{userId}/permissions`. |
| Backend делает | Сохраняет матрицу прав. |
| UI ожидает | Актуальный список permissions. |

Пример запроса:

```json
{
  "permissions": ["documents", "processingLogs", "qa"]
}
```

### Кнопка `Сбросить`

Backend не нужен, если сброс только локально возвращает форму к последнему сохраненному состоянию. Если нужно сбрасывать права до шаблона роли, нужен endpoint:

```http
POST /admin/users/{userId}/permissions/reset
```

### Кнопка `Сохранить изменения`

| Параметр | Описание |
|---|---|
| Endpoint | `PATCH /admin/users/{userId}` и/или `PUT /admin/users/{userId}/permissions`. |
| Backend делает | Сохраняет роль и права, пишет audit-log. |
| UI ожидает | `ok: true`, обновленного пользователя, запись журнала. |

### Административный журнал

| Endpoint | Что возвращает |
|---|---|
| `GET /admin/audit-log?limit=50` | Кто, когда и что изменил: роль, права, статус, документ, системную настройку. |

Пример записи:

```json
{
  "id": "audit-001",
  "time": "2026-05-01T13:10:00+03:00",
  "actor": "Петров А.А.",
  "target": "Иванов С.П.",
  "action": "Изменены роль и права",
  "details": "Роль: Администратор знаний. Доступ: документы, OCR, журналы."
}
```

## 12. Правая панель предпросмотра

Правая панель используется в `Чате`, `Поиске` и `Проверке`.

### Что должен уметь backend

| Возможность | Зачем |
|---|---|
| Открыть одну страницу | Для ссылки `Страница`, чтобы инженер видел конкретное место ответа. |
| Открыть полный документ | Для ссылки `Документ`, чтобы инженер мог пролистать оригинал с первой страницы. |
| Поддержать разные форматы | PDF, DOCX, XLSX, изображения, возможно DWG через конвертацию в PDF/изображение. |
| Подсветить фрагмент | Чтобы было видно, какая часть страницы использована как источник. |
| Отдавать текст страницы | Для копирования, подсветки и проверки OCR. |
| Поддержать resize-панель | UI растягивает панель, backend должен отдавать preview без жесткой привязки к размеру. |

### Рекомендуемые endpoint

| Endpoint | Назначение |
|---|---|
| `GET /documents/{documentId}/file` | Полный документ. |
| `GET /documents/{documentId}/pages/{page}/preview` | Одна страница. |
| `GET /documents/{documentId}/pages/{page}/image` | Растровое изображение страницы. |
| `GET /documents/{documentId}/pages/{page}/text` | OCR-текст страницы. |
| `GET /citations/{citationId}/preview` | Preview по конкретной ссылке из ответа ассистента. |

## 13. Хранилище и обработка документов

Для полной работоспособности UI backend должен иметь не только API, но и слой хранения.

| Слой | Что хранит |
|---|---|
| Файловое хранилище | Оригинальные PDF/DOCX/XLSX/DWG, preview-файлы, изображения страниц. |
| Реляционная БД | Документы, версии, пользователи, роли, права, история, проверки, feedback, audit-log. |
| Поисковый индекс | Полнотекстовый поиск по OCR/parsed text. |
| Векторный индекс | Embeddings чанков для RAG-поиска. |
| OCR/Parsing artifacts | Распознанный текст, структура документа, таблицы, страницы, координаты фрагментов. |
| Очередь задач | OCR, парсинг, индексация, пакетная проверка, экспорт. |

## 15. Что важно согласовать до backend-разработки

| Вопрос | Почему важно |
|---|---|
| Какие форматы документов точно поддерживаем в MVP | От этого зависит preview, OCR, parsing и хранение. |
| Где лежат оригиналы документов | Локальная папка, серверное хранилище, S3-совместимое хранилище или БД. |
| Как версионируются НСИ-документы | Ответы и проверки должны ссылаться на правильную редакцию. |
| Какой OCR/парсер используется | UI показывает OCR-статусы и качество, backend должен считать эти статусы. |
| Нужен ли streaming ответа | Для ChatGPT-подобного поведения лучше SSE/WebSocket, но MVP может работать обычным `POST /chat`. |
| Кто имеет право видеть чужую историю | Влияет на `GET /history` и роли. |
| Что считается полезным ответом | Нужна методика расчета QA-метрики по feedback. |
| Как обрабатываются конфликтующие источники | Backend должен возвращать понятный статус `source_conflict`. |
| Как экспортировать Excel | Клиентским XLSX или backend-файлом с официальным форматированием. |

## 16. Рекомендуемая структура интеграции во frontend

Текущая точка подключения backend: `frontend/src/utils/http.ts`.

Что нужно будет изменить при интеграции:

| Файл | Что сделать |
|---|---|
| `frontend/.env.example` | Указать реальный `VITE_API_BASE_URL`. |
| `frontend/src/utils/http.ts` | Заменить mock fallback на реальные обработчики ошибок и реальные типы ответов. |
| `frontend/src/utils/mockData.ts` | Оставить только для demo/dev-режима или убрать из production-сборки. |
| `frontend/src/store/uiStore.ts` | Подключить пользователя/роль из backend, а не из mock-состояния. |
| `frontend/src/components/*` | Подключить реальные мутации для upload, reindex, OCR, checks, admin-save. |

## 17. Итог

Для полной работоспособности UI backend должен закрыть семь контуров:

1. Пользователи, роли и права.
2. Чат/RAG-ответы с источниками.
3. Поиск по базе знаний.
4. Реестр документов, OCR и индексация.
5. Проверка проектных решений на соответствие НСИ.
6. История, feedback и QA-метрики.
7. Preview страниц и полных документов.

Если эти контуры реализованы, текущий frontend можно подключить к backend без изменения основной структуры экранов.
