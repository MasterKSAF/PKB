Ниже представлено объединённое описание системы, в котором первая (общая) структура дополнена уточнениями из второй части (preview-режимы, легковесные API и взаимодействие с пользователем). Описание согласовано и не содержит противоречий.

## 1. Общая структура и потоки данных



Система состоит из микросервисов, управляемых **оркестратором**. Обработка документа проходит в две возможные фазы:



- **Фаза превью** – быстрый предварительный анализ для получения метаданных и проверки уникальности, с возможностью вмешательства пользователя.

- **Фаза полной обработки** – глубокое извлечение, валидация, сохранение в реестре и индексация.



**Состав сервисов** (все общаются через оркестратор, бинарные данные передаются только по `fileKey`; общее файловое хранилище доступно каждому сервису через единый модуль работы с хранилищем):



- **OCR-сервис** – оптическое распознавание сканов/изображений

- **Parser-сервис** – извлечение из цифровых PDF, DOC, DOCX

- **Сервис конвертации/валидации** – логический парсер, построение иерархии, валидация

- **Registry** – хранение карточек документов и подготовка чанков для индексации

- **RAG Builder** – построение векторного индекса

- **RAG Search** – поиск по индексу



**Оркестратор** управляет пайплайном, журналирует этапы, передаёт `fileKey` между сервисами и реализует двухфазную схему с точкой принятия решения пользователем.

## 2. Сквозные понятия и соглашения



### Bbox (координаты блоков)



- Нормированные координаты (0..1), начало в левом верхнем углу (0,0), ось Y направлена вниз.

- Проходят через всю цепочку обработки, но на этапе индексации могут отбрасываться.

- Подсветка блоков на UI по координатам – опциональная функция, в первую очередь не реализуется.



### FileKey



- Все пути к файлам заменены на `fileKey` (хэш или GUID).

- Модуль файлового хранилища генерирует уникальные ключи для сохраняемых бинарных объектов (изображения, таблицы-картинки и т.д.).

- Сервисы сохраняют/загружают данные только через этот модуль, используя `fileKey` как ссылку.



### Уникальность документа и версионность



- Конкретные правила бизнес-ключа (как считать документы одинаковыми, учёт редакций, префиксы ГОСТ/ГОСТ‑Р/ГОСТ‑ИСО) отложены.

- Механизм проверки заложен через Registry (поиск дубликатов выполняется Оркестратором через `POST /registry/documents/check-uniqueness`).

- Точные правила могут быть доработаны позже, в том числе с использованием LLM.



### Protected spans (запрет разрыва при чанковании)



- Зафиксированы как абстрактное требование: некоторые диапазоны текста или таблиц нельзя разрывать при разбиении на чанки.

- Реализация не обязательна для первого этапа, но учитывается в будущем расширении Registry.



### Карточка документа и превью



- «Зародыш» карточки (превью) формируется на фазе превью на основе первых страниц.

- Полная карточка создаётся после полной конвертации/валидации и сохраняется в реестре.

- Для превью используются те же сервисы, но через специализированные легковесные API.

- Preview-данные (частичный JSON, первичные метаданные, кандидаты в дубликаты) хранятся в Оркестраторе как блок данных в журнале пайплайна и не сохраняются в БД.

### Маппинг полей между этапами

В процессе прохождения документа через пайплайн имена полей могут изменяться.
Ниже приведена таблица соответствия ключевых полей на разных этапах.

| Этап → Этап | Поле (исходное) | Поле (результирующее) | Примечание |
|---|---|---|---|
| OCR/Parser → Converter-validator | `doc_code` | `doc_code` | Сквозное поле, имя не меняется |
| OCR/Parser → Converter-validator | `heading` | `type: text`, извлечение clause из content | Построение иерархии разделов |
| OCR/Parser → Converter-validator | `paragraph` | `type: text`, `content.text` | Прямой перенос |
| OCR/Parser → Converter-validator | `text_block` | `type: textBlock`, `content.block[]` | Rich-форматирование сохраняется объектно |
| OCR/Parser → Converter-validator | `caption` | встраивается в image/table по `linked_number` | Слияние с родительским элементом |
| OCR/Parser → Converter-validator | `image_key` | `type: image`, `content.image_key` | Бинарные объекты → image_key (ссылка на фрагмент изображения) |
| OCR/Parser → Converter-validator | `table` | `type: table`, `content.columns/rows` | Структуризация таблиц |
| OCR/Parser → Converter-validator | `formula` (raw) | `type: formula`, `content.latex` | Извлечение LaTeX |
| OCR/Parser → Converter-validator | `heading_level` | `content.level` | Типографический уровень заголовка → уровень вложенности в `content[]` |
| OCR/Parser → Converter-validator | `list` | `type: list`, `content.items[]` | Структуризация списка |
| OCR/Parser → Converter-validator | `headerFooter` | `type: headerFooter`, `content.text` | Прямой проход |
| Converter-validator → Registry | `doc_code` | `doc_code` | Сквозное поле |
| Converter-validator → Registry | `type` | `type` | Тип секции сохраняется сквозным полем |
| Converter-validator → Registry | `image_key` | `content.image_key` | image_key — ссылка на фрагмент изображения (не file_key) |
| Converter-validator → Registry | `bbox` (сырые) | `bbox` (0..1) | Нормировка координат |
| Registry → RAG Builder / for_rag | `content.image_key` | `content.image_key` | Сквозной проход image_key |
| Registry → RAG Builder / for_rag | `path` (ltree) | `path` | Иерархический путь секции |
| Registry → RAG Builder / for_rag | `section_id` | Registry присваивает каждой секции ID при сохранении. |
| Registry → for_rag | `content.block[]` (textBlock) | `content.text` | Font display details отбрасываются, остаётся только текст для индексации |
| Registry → for_rag | `bbox` | ❌ удаляется | Bbox не нужен для индексации, остаётся только `page` |
| Converter-validator → for_rag | `table` / `list` / `image` / `formula` | `content.markdown` (опционально) | Markdown-представление генерируется для сложных структур. Добавляется при формировании for_rag, в validated_v3 не хранится. |
| Registry → for_rag | `references[].target_doc_code` | `references[].target_document_id` | Добавляется UUID связанного документа из БД (null, если документ ещё не загружен).

> **Примечание:** поле `path` формируется на этапе Converter-validator при построении иерархии.
> В сыром JSON от Parser/OCR (тип `raw_ocr_v4`) это поле может отсутствовать, так как оно
> формируется Converter-validator. В БД хранится как `ltree`. RAG Builder получает его как
> строковое представление пути.

## 3. Описание микросервисов




### 3.1. OCR‑сервис и Parser‑сервис (сервисы распознавания и извлечения сырых данных)



Два независимых сервиса с **единым контрактом выходных данных** (структура сырого JSON). Каждый поддерживает **полный режим** и **preview‑режим**.



#### OCR‑сервис



- **Назначение:** обработка сканов, изображений (JPEG, PNG, TIFF), нефоточитаемых PDF.

- **Вход:** файл (скан/изображение/PDF без текстового слоя).



#### Parser‑сервис



- **Назначение:** разбор цифровых PDF, DOC, DOCX и других офисных форматов с текстовым слоем.

- **Вход:** файл указанного формата.



#### Выход обоих сервисов – сырой JSON



- Блоки плоские, без иерархии.

- Слияние разорванных таблиц при технической возможности

- Определение заголовков/абзацев при технической возможности (учесть в типе)

- LLM не используется.

- Бинарные объекты (картинки, таблицы как изображения) сохраняются в файловое хранилище, возвращается `fileKey`.



#### Режимы работы



**Полный режим:**



- Обрабатываются все страницы, возвращается полный сырой JSON.



**Preview‑режим (легковесный):**



- Обрабатываются только первые N страниц (настраивается).

- Предварительно извлечённый текст ключевых полей простыми эвристиками.

- **Не выполняет** сохранение бинарных объектов в файловое хранилище.

- Используется оркестратором в фазе превью.



### 3.2. Сервис конвертации / валидации (Converter-validator)



Выполняет основную интеллектуальную обработку и также предоставляет **preview API** для фазы превью.



#### Полный режим



- **Вход:** сырой JSON со всех страниц (от OCR или Parser).

- **Выход:** иерархический, типизированный JSON, близкий к итоговому документу-контейнеру.



**Что делает (полный цикл):**



- **Построение иерархии** – преобразует плоские блоки в структуру разделов, подразделов, абзацев, заголовков, списков и т.д. Объединяет таблицы, разорванные на нескольких страницах (по координатам, заголовкам и другим признакам).

- **Извлечение метаданных** – выделяет обозначение документа, наименование, тип, даты, номер редакции и прочее.

- **Распознавание перекрёстных ссылок** – находит упоминания других документов (ГОСТ, ТУ и т.п.) и нормализует их.

- **Использование LLM** – допускается для построения иерархии, классификации типов документов, нормализации структуры и извлечения сложных метаданных.

- **Валидация** – проверка полноты данных, соответствия целевой схеме документа.

- **Вычисление хэшей** – вычисление `file_hash_sha256` и `title_hash_sha256` для последующей проверки уникальности Оркестратором через Registry.

- **Журналирование** – сохраняет промежуточные результаты и ошибки через оркестратор.



> Важно: этот сервис **не сохраняет** никакие данные в БД, он только готовит структурированное представление.



#### Preview API (легковесные конечные точки)



Используются оркестратором на этапе превью. Сервис предоставляет:



1. **Извлечение первичных метаданных**

   

   - **Вход:** сырой JSON первых страниц (из preview-режима OCR/Parser).

   - **Выход:** ключевые атрибуты – обозначение, наименование, предполагаемый тип документа, даты, номер редакции (если найден).

   - Может применять эвристики и/или быстрый вызов LLM с ограничением по времени/токенам.


> Проверка уникальности на preview-фазе выполняется **Оркестратором** после получения метаданных — через `POST /registry/documents/check-uniqueness`. Converter-validator только вычисляет хэши (на full-фазе), но не ищет дубликаты.


Эти методы **не запускают** полную конвертацию, **не сохраняют** никакие данные и работают быстро (важно для интерактивного взаимодействия).



### 3.3. Registry (Сервис реестра документов)



**Назначение:** ведение основной базы документов и подготовка данных для индексации.



- **Вход:** иерархический JSON от сервиса конвертации/валидации (полный режим) с подтверждёнными метаданными.

- **Выход:**

  - Сохранённая карточка документа в реляционной БД.

  - Плоский JSON, содержащий чанки (секции) документа с метаданными для передачи в RAG Builder.



**Что делает:**



- Формирует и сохраняет карточку документа (уникальный ID, обозначение, наименование, тип, срок действия, версия, даты, связи с другими документами).

- **Сегментирование:** разбивает документ на логические **секции** (записи `registry.document_sections`) с привязкой к ID документа.

- Подготавливает «плоский» JSON для RAG Builder: список секций с метаданными, достаточный для построения векторного индекса.

- Дополнительно: предоставляет API для быстрого поиска дубликатов по метаданным (используется preview-методами конвертера-валидатора).



### 3.4. RAG Builder



**Назначение:** построение и обновление векторных индексов для поиска.



- **Вход:** плоский JSON от Registry (сегменты, метаданные, ссылки на `fileKey`).

- **Выход:** готовый к поиску векторный индекс; статус операции.



**Что делает:**



- Принимает сегменты с метаданными.

- **Чанкование:** разбивает **секции** (полученные от Registry) на более мелкие **чанки** для эмбеддингов с привязкой к ID секций и документа. При разбиении учитывает (в будущем) protected spans – неразрывные блоки.

Генерирует эмбеддинги для текстовых фрагментов.

- Сохраняет векторы и связанные метаданные в выбранное векторное хранилище.

- Каждый чанк содержит только текст и ссылки на сегменты для цитирования. Не включает информацию для визуального отображения (полноразмерные картинки, графика, вёрстка).

- Для сложных структур (`table`, `list`, `image`, `formula`) RAG Builder получает готовое `content.markdown` — единое текстовое представление, не требующее дополнительного парсинга. Для `text`/`textBlock`/`headerFooter` используется `content.text`. Markdown генерируется на этапе формирования for_rag (Converter или adapter), а не validated_v3.



### 3.5. RAG Search



**Назначение:** поиск по проиндексированным документам.



- **Вход:** поисковый запрос (текст), опционально фильтры по метаданным, параметры поиска.

- **Выход:** ранжированный список релевантных чанков с идентификаторами документов, текстом, ссылками на исходные файлы (через `fileKey`) и метаданными.



**Что делает:**



- Преобразует запрос в эмбеддинг с той же моделью, что и RAG Builder.

- Выполняет векторный поиск с возможной фильтрацией.

- Возвращает результаты, готовые для использования в LLM-ассистенте или пользовательском интерфейсе.

- (Опционально) логирует запросы для будущего улучшения качества поиска.

## 4. Оркестратор и двухфазный пайплайн



Оркестратор управляет последовательностью вызовов, журналирует каждый этап и обеспечивает передачу `fileKey` вместо файлов. Журнал обработки не входит в реестр документов, но доступен для анализа.



### Фаза превью (с возможностью вмешательства пользователя)

1. Оркестратор определяет тип файла (скан/цифровой) и вызывает **OCR-сервис** или **Parser-сервис** в **preview‑режиме** (обрабатываются только первые N страниц, по умолчанию 3).

2. Полученный частичный сырой JSON передаётся в **сервис конвертации/валидации через preview API**:

   - извлечение первичных метаданных (`POST /converter/preview/metadata`).

3. Оркестратор **самостоятельно** выполняет проверку уникальности: полученные метаданные и `file_size_bytes` (известен с этапа загрузки) передаются в `POST /registry/documents/check-uniqueness`. Размер используется для pre-filtering кандидатов. Это обеспечивает единый механизм duplicate-детекции для preview и full-фазы.

4. Preview-результаты (частичный JSON, метаданные, кандидаты в дубликаты) **сохраняются** в журнале Оркестратора
   (`GET /documents/{doc_id}/history`) с TTL 7 дней для последующего использования в полной фазе.

5. Пользовательский интерфейс отображает:

   - превью (ключевые поля, миниатюры первых страниц);

   - кандидатов в дубликаты.

6. Пользователь принимает решение:

   - `proceed` — продолжить полную обработку;
   - `stop_duplicate` — остановить (документ уже существует);
   - `force_new_version` — принудительно создать новую версию существующего документа.

   (Действия `overwrite` и `force_overwrite` не предусмотрены текущим API; при необходимости расширение действия `force_new_version` может быть дополнено в будущих версиях.)

7. Решение передаётся оркестратору через `POST /tasks/{task_id}/decide`.

8. **Таймаут ожидания решения:** если пользователь не принял решение в течение 24 часов,
   документ автоматически переводится в `failed` (код `DECISION_TIMEOUT`).

9. Повторный вызов `decide` для терминальных статусов (`duplicate`, `new_version`, `archived`)
   возвращает ошибку `400 INVALID_STATE_TRANSITION`.



### Фаза полной обработки (если одобрено)

- **OCR/Parser** в полном режиме → сырой JSON (все страницы).
  При наличии кэшированных preview-результатов полная обработка начинается
  со страницы `max_pages + 1`, исключая повторное распознавание.

- **Converter-validator** (полный режим) → иерархический JSON.
  Preview-метаданные используются как основа, дообогащаются полными данными.
- **Оркестратор** → проверка уникальности через `POST /registry/documents/check-uniqueness` с передачей метаданных и `file_size_bytes` (финальная верификация перед записью).

- **Registry** → сохранение карточки, сегментация на секции, подготовка данных для индексации.

- **RAG Builder** → построение/обновление векторного индекса.

Если проверка уникальности выявила дубликат (на preview или full-этапе), обработка завершается с пометкой «дубликат» или «отклонено»,
без сохранения в реестре и индексации.




**Дополнительные замечания:**



- Все сервисы вызываются оркестратором, который ведёт журнал и управляет ветвлением по результатам превью.

- Файлы и бинарные объекты передаются исключительно через `fileKey` с использованием общего модуля файлового хранилища.

- Координаты `bbox` сохраняются на всех этапах до индексации и могут быть отброшены в чанках для поиска.

- Вопросы точного бизнес-ключа уникальности и `protected spans` оставлены для будущих итераций.
