# PKB RAG Frontend

Frontend для локальной RAG-системы по инженерным документам и нормативно-справочной информации. Интерфейс собран как демонстрационный рабочий контур под ТЗ: чат с источниками, поиск по базе знаний, реестр документов, проверка проектных решений на соответствие требованиям НСИ, история запросов, QA-метрики и администрирование.

Основная текущая версия находится в этой папке и собрана на React + Vite + MUI.

## Что есть в UI

| Раздел | Что делает |
|---|---|
| `Чат` | Основной сценарий работы инженера: вопрос, ответ ассистента, поиск по текущему открытому чату, нумерация выводов, ссылки `Страница` и `Документ`, открытие материалов в правой панели предпросмотра. |
| `Поиск` | Поиск по базе знаний: запрос, фильтры, результаты с релевантностью, фрагментами и открытием страницы или полного документа. |
| `Реестр` | Список документов базы знаний: статусы OCR, индексация, загрузка документов, обновление индекса и переобработка OCR. |
| `Проверка` | Сверка проектного решения с требованиями НСИ: выбор одного или нескольких документов, пакетная проверка, открытие проектных и нормативных источников, выгрузка результата в Excel. |
| `История` | Журнал диалогов: компактные фильтры по пользователю, проекту, теме, статусу и тексту чата, раскрытие найденного диалога прямо внутри таблицы, продолжение выбранного чата и экспорт текущей выборки. |
| `QA` | Контроль качества: метрики OCR, поиска, ответов со страницами, оценка ответов ассистента инженерами и журнал проверки. |
| `Администрирование` | Управление пользователями, ролями и правами доступа. В demo-режиме можно выбрать пользователя, изменить роль, включить/выключить доступы, сохранить изменения и увидеть запись в административном журнале. |
| `Фокус-режим` | Скрывает левую навигацию и оставляет только текущую рабочую область. Доступен на каждой вкладке. |
| `Видеоинструкция` | Полноэкранный demo-сценарий интерфейса. |

## Роли и доступ

Перед входом показывается отдельный экран авторизации в стиле основного интерфейса. В demo-режиме можно выбрать тестового пользователя. После входа в правой верхней части рабочей области отображаются ФИО, должность и роль. Доступные вкладки зависят от роли.

| Роль | Доступные вкладки |
|---|---|
| `Пользователь` | `Чат`, `Поиск`, `Проверка`, `История`. |
| `Администратор` | `Чат`, `Поиск`, `Реестр`, `Проверка`, `История`, `QA`, `Администрирование`. |

В текущей frontend-версии роли работают в demo-режиме через mock-данные. Изменения, сделанные во вкладке `Администрирование`, сохраняются в состоянии интерфейса до перезагрузки страницы и сразу отражаются в таблице пользователей. После подключения backend роль должна приходить из профиля пользователя или auth-сервиса, а изменения прав должны сохраняться на сервере.

Ожидаемая логика доступа к истории диалогов: `Пользователь` видит свои диалоги, `Администратор` видит диалоги всех пользователей для контроля качества, статистики и анализа частых вопросов.

## Темы и режимы

Интерфейс поддерживает две темы:

| Режим | Назначение |
|---|---|
| `Темная тема` | Основной визуальный режим для демонстрации и повседневной работы. |
| `Светлая тема` | Альтернативный режим для светлого рабочего окружения. |

Переключение темы находится в верхней панели рабочей области. Фокус-режим работает независимо от выбранной темы.

## Режим работы

UI может работать без backend в mock/demo-режиме. Это позволяет:

- открыть интерфейс локально;
- пройтись по вкладкам;
- проверить сценарии и кнопки;
- показать продукт команде до готовности API;
- обсудить UX с заказчиком без развернутого backend-контура.

При подключении backend интерфейс использует те же экраны и пользовательские сценарии, но уже с реальными данными.

## Что нужно от backend

Ниже черновой API-контракт для подключения реального backend. Основной принцип простой: frontend отправляет HTTP-запрос, backend возвращает структурированный JSON. Исключение: открытие полного документа или страницы может возвращать файл напрямую, если так удобнее backend-команде.

Названия endpoint пока рабочие. При интеграции их можно заменить в [frontend/src/utils/http.ts](src/utils/http.ts).

### Общие правила обмена

Backend должен отвечать в JSON везде, где не передается файл:

```json
{
  "ok": true,
  "data": {},
  "error": null
}
```

Если произошла ошибка:

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "Документ не найден"
  }
}
```

Все сущности должны иметь стабильные ID: пользователь, документ, страница, ответ, источник, проверка, запись истории.

### Чат инженера

Главный сценарий: инженер задает вопрос, backend ищет релевантные документы и возвращает ответ с источниками.

Запрос:

```http
POST /chat
```

```json
{
  "question": "Какая минимальная толщина листа корпуса?",
  "sessionId": "chat-001",
  "userId": "u-001",
  "context": {
    "projectId": "project-17",
    "documentIds": ["doc-001", "doc-002"],
    "nsiVersion": "2026"
  }
}
```

Обычный ответ:

```json
{
  "ok": true,
  "data": {
    "answerId": "ans-001",
    "status": "answered",
    "message": null,
    "answerItems": [
      {
        "number": 1,
        "text": "Минимальная толщина листа не должна определяться отдельно от проекта. Ее нужно проверять по району корпуса, материалу и расчетной нагрузке.",
        "citations": [
          {
            "citationId": "cit-001",
            "documentId": "doc-001",
            "documentTitle": "Правила классификации и постройки морских судов",
            "section": "Корпус",
            "page": 45,
            "fragment": "Фрагмент текста, на котором основан ответ.",
            "pagePreviewUrl": "/documents/doc-001/pages/45/preview",
            "documentUrl": "/documents/doc-001/file"
          }
        ]
      }
    ],
    "latencyMs": 1420
  },
  "error": null
}
```

Как frontend это показывает:

1. Текст из `answerItems[].text` идет в нумерованный ответ.
2. Под каждым пунктом показываются ссылки `Страница` и `Документ`.
3. `Страница` открывает конкретную страницу справа.
4. `Документ` открывает полный документ с первой страницы.

Если данных не хватает, backend не должен выдумывать ответ. Он возвращает статус `needs_clarification`.

```json
{
  "ok": true,
  "data": {
    "answerId": "ans-002",
    "status": "needs_clarification",
    "message": "Уточните проект, район корпуса и тип судна.",
    "missingFields": ["projectId", "hullArea", "vesselType"],
    "answerItems": []
  },
  "error": null
}
```

Уточнение означает: backend понял вопрос, но не может корректно ответить без дополнительных данных.

Если источники противоречат друг другу, backend возвращает статус `source_conflict`.

```json
{
  "ok": true,
  "data": {
    "answerId": "ans-003",
    "status": "source_conflict",
    "message": "Найдены разные требования в двух редакциях документа.",
    "conflicts": [
      {
        "documentId": "doc-001",
        "documentTitle": "НСИ, редакция 2024",
        "page": 45,
        "value": "8 мм"
      },
      {
        "documentId": "doc-002",
        "documentTitle": "НСИ, редакция 2026",
        "page": 47,
        "value": "10 мм"
      }
    ]
  },
  "error": null
}
```

Конфликт источников означает: backend нашел несколько оснований, но они дают разные значения. Frontend должен показать это как предупреждение, а не как финальный ответ.

Экспорт истории:

```http
GET /history/export?userId=u-001&status=answered
```

Экспорт должен учитывать текущие фильтры. По клику на запись истории frontend раскрывает диалог прямо внутри таблицы истории. Кнопка `Продолжить чат` переносит выбранный диалог на вкладку `Чат`.

Права доступа к истории после подключения backend:

- `Пользователь` получает только свои диалоги.
- `Администратор` может получать все диалоги или выборку по пользователю, проекту, теме и статусу.
- Все действия с историей должны учитывать текущую роль и права пользователя.

Поиск по текущему открытому чату выполняется прямо на вкладке `Чат`. Это отдельный быстрый поиск внутри активной беседы, без перехода в общий журнал истории.

## Демонстрационные сценарии по ТЗ

Для показа заказчику лучше идти не просто по вкладкам, а по рабочим сценариям из ТЗ. Тогда видно, какую задачу решает интерфейс и какой раздел ТЗ закрывается.

| № | Сценарий | Где показываем | Что должно быть видно |
|---|---|---|---|
| 1 | Вход под разными ролями | Экран входа, верхняя панель пользователя, `Администрирование` | У пользователя есть ФИО, должность, роль и разный набор доступных вкладок: пользователь и администратор. |
| 2 | Загрузка и обработка НСИ | `Реестр`, `Администрирование` | Загрузка документа, OCR-статус, индекс, переобработка OCR, обновление базы знаний. |
| 3 | Формирование базы знаний | `Реестр`, `QA` | Документы проходят обработку, получают статус индексации, качество OCR и поиска отражается в метриках. |
| 4 | Поиск по базе знаний | `Поиск` | Инженер вводит запрос, получает список документов, фрагменты, релевантность, ссылки `Страница` и `Документ`. |
| 5 | Диалоговый режим | `Чат` | Инженер задает вопрос, получает нумерованный ответ, при необходимости система должна уметь уточнять контекст. |
| 6 | Ответ с источниками | `Чат`, правая панель предпросмотра | Под каждым пунктом ответа есть `Страница` и `Документ`; страница открывает конкретный фрагмент, документ открывается целиком. |
| 7 | Проверка проектного решения | `Проверка` | Выбор одного или нескольких документов, сверка параметров проекта с НСИ, статусы `OK`, `WARNING`, `ERROR`, комментарий и источники. |
| 8 | Рекомендации и конфликты | `Чат`, `Проверка`, `QA` | Нужно показать, что при конфликте требований система не принимает решение за инженера, а подсвечивает проблему и предлагает проверить источник. |
| 9 | Расчетный сценарий | Не входит в текущий MVP | По ответу заказчика расчетный модуль пока исключен из актуального объема MVP. |
| 10 | История запросов и поиск по чатам | `Чат`, `История` | В текущем чате есть быстрый поиск по сообщениям. В истории хранятся диалоги по пользователям, проектам, темам и сессиям; есть фильтры, просмотр диалога, продолжение чата и экспорт текущей выборки. |
| 11 | QA и критерии приемки | `QA` | Метрики: точность ответов, время ответа, качество OCR, доля ответов с источниками, оценки инженеров и журнал проверки. |
| 12 | Интеграция с ИС «Меридиан» | `История`, `QA`, будущий backend-статус | По ТЗ система должна получать данные через API, записывать результаты и сохранять историю. В текущем UI это требует явной индикации после подключения backend. |

Короткий маршрут демонстрации:

1. Показать экран входа, роль пользователя и доступные ему вкладки.
2. В `Чате` задать вопрос, получить ответ, открыть `Страницу` и `Документ`.
3. В `Поиске` найти документ по базе знаний и открыть найденный источник.
4. В `Проверке` выбрать документы, запустить сверку, показать статусы и экспорт в Excel.
5. Войти под администратором и показать `Реестр`.
6. В `Администрировании` показать пользователей, роли и права доступа.
7. В `Истории` показать сохраненные диалоги, компактные фильтры, раскрытие найденного чата внутри таблицы, продолжение чата и экспорт.
8. В `QA` показать метрики приемки и оценки ответов инженерами.
9. Отдельно проговорить сценарии, которые требуют backend или дальнейшей доработки: интеграция с ИС «Меридиан», реальные документы из БЗ и хранение истории диалогов.

| Направление | Статус |
