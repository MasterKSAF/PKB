# PKB RAG Frontend

Frontend для локальной RAG-системы по инженерным документам и нормативно-справочной информации. Интерфейс собран как демонстрационный рабочий контур под ТЗ: чат с источниками, поиск, реестр документов, проверка на соответствие требованиям НСИ, история запросов и QA-блок.

## Версии frontend

| Папка | UI-библиотека | Назначение |
|---|---|---|
| **[frontend-v1](../frontend-v1)** | MUI | Основная текущая версия UI. |
| **[frontend-v2](../frontend-v2)** | Не выбрана | Заготовка под альтернативную версию UI. На текущий момент полноценная реализация не добавлена. |

## Что есть в UI

| Раздел | Что делает |
|---|---|
| `Чат` | Основной сценарий работы инженера. Ответы показываются вместе со ссылками `Страница` и `Документ`, а сами страницы и документы открываются в правой панели предпросмотра. |
| `Поиск` | Поиск по базе знаний с фрагментами, релевантностью и открытием страницы или полного документа. |
| `Реестр` | Список документов базы знаний, OCR-статусы, индекс, действия по загрузке и обновлению. |
| `Проверка` | Сопоставление проектного решения с требованиями НСИ. Есть выбор документов, пакетная проверка, открытие страниц и документов, выгрузка результата в Excel. |
| `История` | Журнал запросов, ответов, пользователей и статусов с фильтрами и экспортом. |
| `QA` | Контрольные метрики, оценка ответов ассистента и журнал проверки. |
| `Фокус-режим` | Доступен на каждой вкладке и скрывает левую панель, чтобы оставить только текущую рабочую область. |
| `Видеоинструкция` | Полноэкранный demo-сценарий интерфейса. |

## Режим работы

UI может работать без backend в mock/demo-режиме. Это позволяет открыть интерфейс локально, пройтись по вкладкам, проверить сценарии и кнопки, а также показать продукт команде до готовности API.

При подключении backend интерфейс использует те же экраны и ту же структуру пользовательского сценария, но уже с реальными данными.

## Docker-запуск

Из папки `UI-UX/frontend-v1`:

```bash
docker compose up --build
```

После запуска открыть:

```text
http://localhost:3000
```

Остановить контейнер:

```bash
docker compose down
```

## Что нужно от backend

Backend должен работать локально или в контуре заказчика и отдавать данные в JSON. UI не должен сам искать по документам, считать embeddings, выполнять OCR или принимать инженерные решения. Задача frontend — отправить запрос, показать ответ, источники, статусы и результаты проверки в понятном для инженера виде.

### Общие требования к API

| Что нужно | Зачем это нужно в UI |
|---|---|
| Единый JSON-формат ответов | Чтобы frontend стабильно отображал данные без ручной обработки разных форматов. |
| Стабильные `id` для документов, страниц, чанков, запросов и ответов | Чтобы UI мог открывать источники, хранить историю и отправлять feedback к конкретному ответу. |
| Понятные статусы операций | Чтобы UI показывал `loading`, `success`, `warning`, `error`, `needs_clarification`, а не угадывал состояние по тексту. |
| Тексты ошибок для пользователя и технические коды ошибок | Чтобы инженер видел понятное сообщение, а разработчики могли отлаживать проблему. |
| URL или endpoint для просмотра страницы и полного документа | Чтобы ссылки `Страница` и `Документ` открывали правильное превью справа. |
| Время обработки и служебные метаданные | Чтобы вкладка `QA` могла показывать latency, качество поиска, OCR и ошибки. |

### Чат и ответ ассистента

Backend должен принимать вопрос инженера и возвращать структурированный ответ, а не просто длинный текст.

| Поле | Смысл |
|---|---|
| `answerId` | ID ответа, нужен для истории и feedback. |
| `message` | Общий текст ответа или краткое пояснение. |
| `items` | Массив пунктов ответа. Лучше, если каждый пункт ответа связан со своим источником. |
| `items[].text` | Текст конкретного пункта ответа. |
| `items[].sources` | Источники для конкретного пункта: документ, страница, раздел, фрагмент. |
| `confidence` | Уверенность или оценка качества ответа, если backend ее считает. |
| `status` | `success`, `needs_clarification`, `conflict`, `error`. |
| `clarificationQuestion` | Вопрос инженеру, если данных недостаточно. |

Для каждого источника нужны минимум:

| Поле | Смысл |
|---|---|
| `documentId` | ID документа. |
| `documentTitle` | Название документа для подписи в UI. |
| `page` | Номер страницы или диапазон страниц. |
| `section` | Раздел документа, если он есть. |
| `fragment` | Короткий текстовый фрагмент, на котором основан ответ. |
| `pagePreviewUrl` или `pagePreviewId` | Что открывать по ссылке `Страница`. |
| `documentPreviewUrl` или `documentId` | Что открывать по ссылке `Документ`. |

### Поиск

Для вкладки `Поиск` backend должен отдавать список найденных фрагментов с релевантностью и ссылками на просмотр.

| Поле | Смысл |
|---|---|
| `query` | Исходный поисковый запрос. |
| `results[]` | Найденные документы или фрагменты. |
| `results[].documentTitle` | Название документа. |
| `results[].page` | Страница источника. |
| `results[].section` | Раздел документа. |
| `results[].fragment` | Фрагмент найденного текста. |
| `results[].score` | Релевантность или ранжирование. |
| `results[].pagePreviewId` | Открытие конкретной страницы. |
| `results[].documentId` | Открытие полного документа. |

### Реестр документов

Для вкладки `Реестр` backend должен отдавать список документов базы знаний и статусы обработки.

| Поле | Смысл |
|---|---|
| `documentId` | ID документа. |
| `title` | Название документа. |
| `type` | Формат: PDF, DOCX, scan, CAD, HTML/XML и т.д. |
| `pages` | Количество страниц, если применимо. |
| `ocrStatus` | Статус OCR: не требуется, в работе, готово, ошибка. |
| `indexStatus` | Статус индексации: не индексирован, индексируется, готов, ошибка. |
| `updatedAt` | Дата последней обработки. |
| `warnings` | Замечания: плохой OCR, битая структура, не найден текст и т.д. |

### Проверка на соответствие НСИ

Для вкладки `Проверка` backend должен принимать выбранные проектные документы и отдавать результат сопоставления проектного решения с требованиями НСИ.

| Поле | Смысл |
|---|---|
| `checkId` | ID проверки. |
| `project` | Проект или документ, который проверяется. |
| `section` | Раздел проектной документации. |
| `parameter` | Проверяемый параметр. |
| `projectValue` | Значение, найденное в проекте. |
| `nsiRequirement` | Требование НСИ, с которым идет сверка. |
| `nsiDocument` | Документ НСИ, где найдено требование. |
| `status` | Результат проверки: `OK`, `WARNING` или `ERROR`. |
| `comment` | Пояснение для инженера. |
| `sources` | Ссылки на страницу и документ НСИ. |
| `exportUrl` или файл ответа | Данные для выгрузки результата в Excel. |

### История

Для вкладки `История` backend должен хранить и отдавать обращения пользователей.

| Поле | Смысл |
|---|---|
| `requestId` | ID запроса. |
| `user` | Пользователь или роль. |
| `question` | Текст запроса. |
| `answerSummary` | Краткий ответ или статус ответа. |
| `sources` | Использованные источники. |
| `createdAt` | Дата и время запроса. |
| `status` | Успешно, ошибка, требует уточнения, ручная проверка. |
| `latencyMs` | Время обработки. |

### Feedback и QA

Для оценки ответов frontend должен отправлять feedback по конкретному `answerId`, а backend должен сохранять эти оценки и учитывать их в QA.

| Поле | Смысл |
|---|---|
| `answerId` | Какой ответ оценили. |
| `requestId` | К какому запросу относится ответ. |
| `rating` | Полезно / не полезно или другая согласованная шкала. |
| `comment` | Комментарий инженера. |
| `user` | Кто оставил feedback. |
| `createdAt` | Когда оставили feedback. |

Для вкладки `QA` backend должен отдавать:

| Метрика | Что показывает |
|---|---|
| `answerAccuracy` | Оценка качества ответов по тестам или ручной разметке. |
| `sourceCoverage` | Доля ответов, где есть источники. |
| `ocrQuality` | Качество OCR по обработанным документам. |
| `avgLatencyMs` | Среднее время ответа. |
| `positiveFeedbackRate` | Доля положительных оценок инженеров. |
| `manualReviewQueue` | Сколько ответов требует ручной проверки. |
| `logs[]` | Последние события: ошибки поиска, OCR, конфликт источников, успешные проверки. |

## Основные API

| Метод | Endpoint | Для чего используется |
|---|---|---|
| `POST` | `/chat` | Отправка вопроса инженера и получение ответа ассистента. |
| `GET` | `/search` | Поиск по базе знаний. |
| `GET` | `/documents` | Получение реестра документов. |
| `GET` | `/source-preview` | Получение превью страницы или полного документа, если backend отдает превью отдельным endpoint. |
| `GET` | `/checks` | Получение результатов проверки НСИ. |
| `GET` | `/history` | Получение истории запросов и ответов. |
| `POST` | `/feedback` | Отправка оценки ответа инженером. |
| `GET` | `/metrics` | Получение QA-метрик и журнальных записей. |

## Демонстрационный сценарий

1. Инженер открывает `Чат` и задает вопрос по документам или нормам.
2. Система возвращает ответ и ссылки `Страница` и `Документ`.
3. Инженер открывает нужную страницу или полный документ в правой панели.
4. При необходимости инженер переходит в `Поиск`, `Проверку` или `Реестр`.
5. Тимлид или аналитик смотрит `История` и `QA`, чтобы оценить качество работы системы.
