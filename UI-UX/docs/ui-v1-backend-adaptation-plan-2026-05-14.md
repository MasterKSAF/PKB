# UI V1: план адаптации под backend API

Дата обновления: 2026-05-14

Основа: `docs/api`, `docs/ui_compare.md`, ответы backend-команды на вопросы UI

## 1. Что backend уже подтвердил

Эти пункты считаем рабочими решениями и больше не держим как открытые вопросы.

### Единый вход в API

Frontend ходит в единый публичный API:

```text
VITE_API_BASE_URL=https://{host}/api/v1
```

Для локального запуска:

```text
VITE_API_BASE_URL=http://localhost:{port}/api/v1
```

UI не должен напрямую выбирать между `Orchestrator`, `Query`, `Registry`, `Auth` и другими сервисами. Для frontend это один API-контур `/api/v1`, а маршрутизация по сервисам остается задачей backend.

### Роли и права

Backend предлагает три технические роли:

| Роль в API | Отображение в UI | Смысл |
|---|---|---|
| `engineer` | Инженер | Чат, поиск, проверка, история |
| `knowledge_admin` | Администратор НСИ | Все выше + загрузка/удаление документов, CRUD реестра, мониторинг |
| `system_admin` | Системный администратор | Полный доступ, пользователи, роли, аудит, реестры |

Вкладки и действия в UI берем не из ручной логики frontend, а из ответа `GET /auth/me`:

```json
{
  "role": "engineer",
  "role_title": "Инженер",
  "available_tabs": ["chat", "search", "checks", "history"],
  "permissions": {
    "can_upload_documents": false,
    "can_run_ocr": false,
    "can_manage_users": false,
    "can_manage_classifiers": false,
    "can_manage_terminology": false,
    "can_manage_registry": false
  }
}
```

### Чат

Основной сценарий для UI: `POST /chat`.

Если `session_id` не передан, backend создает чат-сессию автоматически. Явное создание через `POST /chat/sessions` остается дополнительным сценарием.

### История

Главный экран истории лучше строить вокруг чат-сессий:

- `GET /chat/sessions` - список сессий;
- `GET /chat/sessions/{session_id}` - полный диалог внутри сессии;
- `GET /chat/history` - плоский журнал отдельных вопросов и ответов.

В UI основной сценарий: открыть список сессий, раскрыть нужный диалог, при необходимости продолжить чат.

### Поиск по старым чатам

Сейчас backend описывает:

- поиск сессий по `title`: `GET /chat/sessions?search=<текст>`;
- фильтрацию плоской истории: `GET /chat/history?user_id=&status=&date_from=&date_to=`.

Полноценный full-text поиск по содержимому всех сообщений пока не описан.

### Документы: проектные и НСИ

Разделение идет по `document_type`:

| `document_type` | Категория | Назначение |
|---|---|---|
| `normative` | НСИ | ГОСТ, Правила РС и другие нормативные документы |
| `archival_scan` | НСИ | Архивные сканы |
| `drawing` | Проектный документ | Чертежи |
| `specification` | Проектный документ | Спецификации |

Для проверки НСИ backend ожидает раздельные списки:

```json
{
  "project_document_ids": ["doc-project-001"],
  "nsi_document_ids": ["doc-nsi-001", "doc-nsi-002"],
  "parameters": ["толщина листа", "марка стали"]
}
```

`user_id` берется из токена, а не передается из UI.

### Загрузка документов

Главный endpoint загрузки:

```text
POST /documents
```

Загрузка асинхронная. После нее backend запускает OCR, парсинг структуры и индексацию. Статус отслеживается через:

```text
GET /documents/{doc_id}/status
```

### `/documents` и `/registry/documents`

Это разные сущности, их нельзя смешивать.

| API | Что означает |
|---|---|
| `/documents` | Физический файл: загрузка, OCR, страницы, индекс, ошибки обработки |
| `/registry/documents` | Карточка НСИ-документа: название, номер, статус, классификатор, метаданные |

Связь должна идти через:

```text
documents.metadata.registry_doc_id -> registry.documents.doc_id
```

### Registry Service

Registry Service логически относится к вкладке `Реестр` / `НСИ` и административным функциям.

Для MVP backend закладывает:

- просмотр;
- создание;
- обновление;
- изменение статуса;
- удаление;
- импорт;
- экспорт.

Это относится к:

- НСИ-документам;
- классификаторам;
- терминологии.

## 3. Что пока подсвечиваем как открытые зоны

### Preview DOCX/XLSX/PPTX

Backend подтвердил, что поддержка DOCX/XLSX/PPTX для preview пока не описана.

В UI оставляем универсальный просмотр по ссылке/URL, но не обещаем корректный предпросмотр каждого офисного формата.

### Полнотекстовый поиск по чатам

Сейчас есть поиск сессий по заголовку и фильтрация истории. Full-text поиск по содержимому всех сообщений не описан.

Текущий поиск по открытому чату можно оставить на frontend, но поиск по старым чатам требует отдельного решения backend.

### Проверка НСИ

Endpoint `POST /validate/checks` обозначен, но backend пишет, что функционал сверки пока не проработан.

UI может показать сценарий, форму выбора документов и таблицу результата, но точная логика проверки остается открытой.

### Формат `POST /chat`

Backend подтвердил, что `POST /chat` основной для UI, но нужно закрепить финальный формат тела запроса и ответа.

Особенно важны поля:

- текст вопроса: `question`, `content` или другое имя;
- структура пунктов ответа;
- структура источников;
- `session_id`;
- `answer_id` / `message_id`;
- статусы `needs_clarification` и `source_conflict`.

### История для администратора

Не до конца ясно, видит ли администратор:

- только свои чаты;
- чаты всех пользователей;
- чаты только в пределах проекта/роли;
- обезличенную статистику без содержимого.

### Registry в навигации

Backend говорит, что для `engineer` вкладка `Реестр` не отображается, но чтение `/registry/*` может быть разрешено.

Нужно решить UI-поведение:

- скрывать вкладку полностью для инженера;
- показывать НСИ только как фильтр/источник в поиске;
- показывать реестр только администраторам.

## 4. Актуальный список вопросов backend-команде

Этот список нужен для рабочей встречи с backend-командой. Вопросы сформулированы не только как "что спросить", но и как "зачем это нужно UI", чтобы быстрее согласовать контракт без лишних кругов уточнений.

### 1. Какие API уже готовы для UI V1

Нужно понять, какие части интерфейса можно подключать к реальному backend уже сейчас:

- вход в систему;
- чат;
- история чатов;
- поиск по документам;
- предпросмотр страницы и полного документа;
- проверка НСИ;
- обратная связь по ответам;
- метрики качества.

Это поможет разделить UI на две части: что подключаем сразу, а что пока оставляем на демонстрационных данных.

### 2. Авторизация и роли пользователей

Нужно понять, как пользователь входит в систему: логин, пароль и получает ли он временный ключ доступа для дальнейших запросов.

Под временным ключом доступа имеется в виду токен: строка, которую frontend прикладывает к последующим запросам, чтобы backend понимал, кто делает запрос и какие действия этому пользователю разрешены.

После входа UI должен понять, кто вошел:

- инженер;
- администратор НСИ;
- системный администратор.

От этого зависит, какие вкладки и кнопки показывать. Например, инженеру не нужно управление пользователями, а системному администратору оно нужно.

### 3. Профиль пользователя после входа

Нужно понять, что backend возвращает после входа или по запросу профиля пользователя:

- ФИО;
- должность;
- роль;
- доступные вкладки;
- разрешенные действия.

Это нужно не для внешнего оформления, а для управления доступом. Frontend не должен сам решать, кому что можно. UI получает от backend список вкладок и прав, а затем показывает или скрывает соответствующие функции.

Например:

- если backend не вернул право на загрузку документов, UI скрывает или блокирует кнопку загрузки;
- если backend не вернул вкладку администрирования, UI не показывает ее пользователю;
- если пользователь может только читать реестр, UI не показывает действия редактирования.

### 4. Чат инженера

Нужно зафиксировать точный формат обмена: что UI отправляет в backend и что получает обратно.

Со стороны UI обычно отправляется:

- текст вопроса;
- идентификатор текущего диалога, если он уже есть;
- при необходимости выбранные документы или контекст.

Со стороны backend ожидается:

- идентификатор диалога;
- идентификатор ответа;
- статус ответа;
- текст ответа;
- пункты ответа;
- список источников;
- ссылки на страницу и документ.

Это нужно, чтобы UI мог корректно связать вопрос, ответ, источники и обратную связь инженера. Без этого будет трудно понять, к какому именно ответу относится оценка пользователя или какая цитата подтверждает конкретный пункт ответа.

### 5. Источники, страницы и документы

Нужно понять, что backend будет отдавать по каждой ссылке:

- конкретную страницу;
- полный документ;
- PDF;
- изображение страницы;
- оригинальный файл;
- подготовленное preview.

Это важно для правой панели предпросмотра. В UI есть две разные сущности:

- `Страница` открывает конкретный фрагмент или страницу, на которую ссылается ответ;
- `Документ` открывает полный документ с возможностью просмотра и прокрутки.

Нужно заранее договориться, что именно frontend получит от backend, чтобы кнопки `Страница` и `Документ` работали одинаково во всех вкладках.

### 6. История чатов и продолжение диалога

Нужно понять, как backend хранит диалоги:

- отдельными сессиями;
- по пользователям;
- по проектам;
- по темам;
- по рабочим задачам.

Это нужно, чтобы инженер мог открыть старый разговор, продолжить его и не потерять контекст. Также важно понять, что делать с текущим открытым чатом, если пользователь решил продолжить старый диалог из истории.

Нужно отдельно согласовать:

- создается ли новый чат при продолжении старого;
- перезаписывается ли текущий активный чат;
- можно ли продолжить старый чат как новую копию;
- какие ограничения доступа действуют для чужих чатов.

### 7. Поиск по старым чатам

Заказчик отметил, что поиск по чатам желателен. Нужно уточнить, будет ли backend искать по тексту старых вопросов и ответов или только по названию диалога и фильтрам.

Это нужно для нескольких сценариев:

- инженер хочет найти ранее полученный ответ;
- тимлид хочет посмотреть повторяющиеся вопросы;
- команда хочет собирать статистику частых вопросов;
- система в будущем может использовать такие данные для улучшения ответов.

Если полноценного поиска по содержимому старых чатов пока нет, то в UI оставляем поиск по текущему открытому чату и фильтрацию истории.

### 8. Поиск по базе документов

Нужно уточнить, какой запрос UI должен отправлять для поиска по базе знаний и какие фильтры доступны:

- проект;
- тип документа;
- дата;
- НСИ;
- раздел;
- статус документа.

Backend должен возвращать понятный для интерфейса результат:

- документ;
- страницу;
- фрагмент текста;
- релевантность;
- ссылку на страницу;
- ссылку на полный документ.

Это нужно, чтобы вкладка `Поиск` и источники в чате работали по одной логике и пользователь видел одинаковое поведение ссылок.

### 9. Проверка проектного решения по НСИ

Нужно понять, как UI должен отправлять документы на проверку:

- один проектный документ против НСИ;
- несколько проектных документов против НСИ;
- выбранные документы НСИ;
- все актуальные документы НСИ;
- выбранные параметры проверки.

Backend должен вернуть строки результата:

- проект;
- раздел;
- параметр;
- значение в проекте;
- требование НСИ;
- статус;
- комментарий;
- ссылки на источники.

Это нужно для вкладки `Проверка`, чтобы таблица не была просто демонстрационной, а могла показывать реальный результат сверки. Также нужно уточнить, кто формирует Excel-выгрузку: frontend или backend.

### 10. Реестр и загрузка документов

Нужно разделить две сущности:

- физический файл;
- карточка документа НСИ.

Физический файл проходит загрузку, OCR, парсинг и индексацию. Карточка НСИ содержит название, номер, статус, классификатор и метаданные.

UI должен понимать:

- откуда брать статус обработки файла;
- откуда брать описание документа;
- где показывать ошибки OCR;
- где показывать статус индексации;
- как связать загруженный файл с карточкой НСИ.

Это важно для вкладок `Реестр` и `Администрирование`, чтобы не смешать техническую обработку файла и нормативную карточку документа.

### 11. QA и обратная связь инженеров

Нужно понять, какие метрики backend сможет отдавать для вкладки QA:

- качество ответов;
- оценки инженеров;
- скорость ответа;
- ошибки;
- вопросы без ответа;
- качество поиска источников;
- качество OCR;
- журнал проверки.

Также важно понять, что происходит после оценки ответа инженером:

- просто сохраняется комментарий;
- меняется статистика качества;
- создается задача на ручную проверку;
- ответ попадает в выборку для последующего анализа.

Это нужно, чтобы блок QA был не декоративным, а показывал реальные показатели работы системы.

### 12. Ошибки и недоступность backend

Нужно договориться, как backend сообщает об ошибках:

- нет доступа;
- документ не найден;
- не хватает данных;
- конфликт источников;
- сервис временно недоступен;
- истекла авторизация;
- ошибка обработки документа.

Это нужно, чтобы UI показывал человеку понятное сообщение, а не техническую ошибку. Например, инженер должен видеть не `500 Internal Server Error`, а понятное объяснение: "Сервис поиска временно недоступен, попробуйте повторить запрос позже".

## 5. Приоритет внедрения

### Сначала

1. Обновить `http.ts` под `/api/v1`, token и разные форматы ответа.

## 6. Короткий вывод

UI V1 оставляем базовой рабочей версией. По ответам backend уже можно уверенно уточнить роли, единый API-контур, чат-сессии, историю, загрузку документов и разделение `/documents` с `/registry/documents`.

Главные открытые риски: проверка НСИ, preview офисных файлов, full-text поиск по старым чатам и точный формат ответа `POST /chat`.
