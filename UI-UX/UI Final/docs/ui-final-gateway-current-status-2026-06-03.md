Дата проверки: 03.06.2026
UI-ветка: `feature/ui-final-gateway-current`
UI-папка: `UI-UX/UI Final/frontend`
Gateway-ветка: `origin/develop_gateway`
Gateway-коммит: `cf6eeec376d7c445a3febf63b5e2c1d973b07431`
Базовый URL для UI: `http://127.0.0.1:8081/api/v1`

## Краткий вывод

Интеграция с актуальным Gateway в целом работоспособна. Единый Gateway теперь стартует, backend-тесты проходят, а UI Final в prod-режиме получает ответы чата, поиск, документы, историю, метрики QA и административные данные.

Главные ограничения сейчас не в запуске Gateway, а в неполном совпадении продуктовых сценариев UI и текущих контрактов Gateway:

- для вкладки `Проверка` нет отдельного Gateway API сверки проектных параметров с требованиями НСИ;
- в Gateway нет отдельной сущности `Проект` для дерева `Мои проекты / чаты`, есть только `chat sessions`;
- часть административных и справочных API уже есть в Gateway, но UI пока использует только базовый минимум.

## Что изменено в UI

- Дерево `Мои проекты / чаты` в prod-режиме теперь загружает чаты из `GET /chat/sessions`.
- Выбор чата загружает сообщения через `GET /chat/sessions/{session_id}`.
- Создание, переименование и удаление чата синхронизируются с `POST /chat/sessions`, `PUT /chat/sessions/{session_id}`, `DELETE /chat/sessions/{session_id}`.
- При первом Gateway-запросе UI синхронизирует текущего пользователя через `GET /auth/me`, чтобы локальный пользователь не заменялся ошибочно первым пользователем из Gateway.
- Превью страницы источника теперь запрашивает не только `pages/{page}/preview`, но и `pages/{page}/text`.

## Что работает

1. Gateway как единая точка входа запускается на `8081`.
2. UI Final может работать в prod-режиме против `http://127.0.0.1:8081/api/v1`.
3. Чат, поиск, база знаний, история, QA и часть администрирования уже подключены к Gateway.
4. При недоступности Gateway UI не падает, а уходит в fallback/demo-сценарии.

## Что не работает или остается неполным

1. Вкладка `Проверка` не может быть честно подключена, потому что в Gateway нет endpoint для сценария “проектное решение против требований НСИ”.
2. `Мои проекты` в левой навигации пока не являются backend-сущностью. Gateway отдает chat-сессии, но не проекты, поэтому проекты остаются UI-группировкой.
3. `GET /system/health` есть в нескольких подключенных сервисах. Gateway работает, но OpenAPI предупреждает о дублирующихся operation id.
4. Многие справочные API уже есть, но UI пока не использует их полностью: terminology, registry CRUD, classifier import/quarantine/validate, roles CRUD.

## Как исправить оставшееся

1. Для `Проверки` добавить в Gateway отдельный контракт:
   - запуск проверки выбранных документов/параметров;
   - получение статуса проверки;
   - получение таблицы результатов;
   - открытие источников по строке результата;
   - экспорт результата.
2. Для дерева чатов решить, нужна ли backend-сущность `project/workspace`:
   - если нужна, добавить CRUD проектов и связь `project_id -> chat sessions`;
   - если не нужна, UI должен показывать только список chat-сессий без управления проектами.
3. Для ролей и прав договориться, что является источником истины:
   - список ролей из `GET /admin/roles`;
   - права из `GET /auth/me`;
   - доступные вкладки из `available_tabs`.
4. Для health endpoint оставить один публичный агрегированный путь, например `GET /api/v1/system/health`, а сервисные health развести по именованным путям.
5. Для справочников решить, какие функции действительно нужны в UI Final, чтобы не переносить всю backend-админку без продуктовой необходимости.
