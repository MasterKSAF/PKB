## 📋 Общие результаты

### 1. Архитектура сервисов и распределение зон ответственности

Была подробно обсуждена микросервисная архитектура проекта. Ключевые решения:

- **Запрет перекрёстных зависимостей между сервисами** — каждый сервис полностью автономен внутри себя. Если сервису нужна дополнительная обработка (например, OCR — очистка изображений), он делает это сам, без выноса в оркестратор.
- **Каждый сервис имеет свой микропайплайн** внутри себя. Оркестратор не управляет внутренней логикой сервисов.
- **Query-сервис** теперь отвечает за полный pipeline обработки чата (включая взаимодействие с LLM), он убран из оркестратора. Чат сам является мини-очередью.
- **Gateway** — на проде будет Nginx, который сам разруливает запросы и проверяет авторизацию. Ни один сервис, кроме веба, с авторизацией не взаимодействует.
- **Взаимодействие** между сервисами — через Long Pooling.

### 2. JSON-схема

- Финальное API по карточке документа почти готово, будет выложено в ближайшее время (в течение 2 часов после встречи).
- Добавлены `task_id` (ID задачи) и `file_hash`/`title_hash` (для контроля уникальности контента).
- Парсер возвращает примитивную разметку (страницы, блоки), **Validator** обогащает её в иерархическую структуру, **Registry** расщепляет и сохраняет в БД с `section_id`/`parent_id`, а **REG** получает уже плоский текст для индексации.
- Для REG таблицы предложено передавать в виде простой матрицы (без объектной сложности), т.к. для поиска важны только текст и цифры.

### 3. Взаимодействие OCR и Парсинга (гибридные документы)

- Если в документе есть и текст, и изображения — оркестратор (pipeline-сервис) будет последовательно передавать документ сервисам. Каждый сервис в ответе сообщает, что он смог обработать, а что нет.
- Полноценная обработка гибридных документов — доработка на будущее. Пока решили работать с контрастными типами: скан → OCR, цифровой документ → парсинг.

### 4. Мониторинг (презентация Дианы)

Диана представила пробную версию системы мониторинга:
- **Стек**: OpenTelemetry Collector + Prometheus + Victoria Metrics + Loki + Grafana + Docker Swarm
- Система собирает три типа данных: **трейсы**, **метрики**, **логи**
- Всё развёрнуто и протестировано локально на тестовом ноутбуке
- **Предложение по упрощению**: рассмотреть связку OpenTelemetry + Elasticsearch + Jaeger/Kibana как более компактную альтернативу (Elasticsearch может занять место Loki, а метрики — вопрос второстепенный, приоритет — логи).
- **Планы**: изучить совместимость Jaeger/Elasticsearch с метриками, упростить схему, написать публичную библиотеку для сервисов (middleware).
- Подготовлены инструкции по подключению сервисов к системе мониторинга.
