## 🧭 Ключевые темы обсуждения

### 1. Архитектура сервисов — основное решение встречи
Принято решение **разделить монолитный парсер на два параллельных микросервиса**:

| Сервис | Назначение | Ответственный |
|---|---|---|
| **OCR-сервис** (гибридный) | Распознавание изображений, сканов, чертежей, картинок + текстовый слой | На базе наработок первой группы; Дмитрий исследует инструменты |
| **Парсер текстов** | Парсинг PDF, DOCX и других текстовых форматов, извлечение текста, таблиц, структуры | Диана — обёртка каркаса с заглушками |

**Формат данных:** Упрощённая JSON-структура: документ → страницы → блоки (с типом, содержимым, координатами). Таблицы, формулы и изображения — специальные типы блоков. Полноценная иерархия и анализ — задача конвертера/валидатора.

### 2. Инструменты
- **Дмитрий** продемонстрировал работу `Marker PDF` и `pdf.to.md` — хорошо извлекают текст, формулы (в TeX), таблицы (даже без границ), чертежи. Требуют GPU для сложных случаев.
- **Диана** предлагает `Open Data Loader` — хорошо показал себя в её экспериментах (журнал экспериментов №3).
- **Сергей** поделился опытом: Marker PDF на сканах даёт ~30% ошибок в структуре. Рекомендует подход: извлекать текст отдельно, остальное помечать кандидатами (таблица, картинка) и записывать в JSON.

### 3. Оркестрация сервисов
- Сервисы работают **последовательно-параллельно**: первый (текстовый парсер) пробует обработать документ; если не может — передаёт OCR-сервису.
- Конвертер собирает из блоков единый документ.
- **FastAPI** — API-прослойка.
- **Docker Compose** — для сервера (изоляция окружения). Для локальной разработки — без Docker, чтобы ускорить итерации тестирования.

### 4. Инфраструктура (GPU)
- **Сервера с GPU пока нет**, только CPU.
- Обсуждались бюджетные варианты: AMD-карта 16 ГБ (~25 тыс. руб.), майнинговые карты (~3,5 тыс. руб.).
- Для OCR нужна **локальная маленькая LLM** (много мелких запросов). Для сложных задач — **внешняя LLM**.
- Вопрос сервера решается через администратора оборудования; будет уточняться.
