## 2. Принятые технические решения
- **Эмбеддинги**: решили делать под OpenAI‑совместимый API (чтобы можно было подключать и локальные модели типа Ollama/Qwen, и внешние). Качество важнее, чем бесплатные маленькие модели (они не обеспечивают нормального поиска).
- **База данных**: PostgreSQL 15/16 + pgvector. Схемы: «чистилище» и основная БД. Докер‑контейнеры для каждого сервиса.
- **Мониторинг и логи**: будет централизованный сбор логов, метрик и трейсов (Prometheus + Grafana). Договорились использовать единый подход (возможно, библиотеку `loguru`). Диана подготовит упрощённую версию.
- **Тесты**: обязательны юнит‑тесты для каждого сервиса (желательно с возможностью локального запуска через скрипт). Это поможет нейронкам (Cursor и пр.) отлаживать код.

## 3. Основные проблемы / риски (зафиксированы в таблице)
- Реализации ключевых сервисов до сих пор нет (только методики и наработки).
- Не хватает разработчика на сервис индексации/поиска (Виталий пока не подтвердил участие).
- Неопределённость с мощностью железа (видеокарта для локальных эмбеддингов). Если видеокарты не будет – придётся использовать платные облачные модели.
- При смене модели эмбеддингов или параметров чанков потребуется полная переиндексация (это трудоёмко).
