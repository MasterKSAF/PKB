# OKB RAG Frontend

Frontend для локальной RAG-системы инженерного конструкторского бюро. Интерфейс сделан под требования ТЗ: ввод запроса, диалог, ответ с источниками, история, реестр документов, сверка параметров и QA-контур.

## Что есть в UI

- `Чат`: основной диалог в стиле ChatGPT, уточняющие вопросы, статус ответа, источники прямо под ответом, просмотр источника в отдельном окне.
- `Поиск`: поиск по базе знаний с фильтрами, фрагментами и релевантностью.
- `Реестр`: список документов, OCR-статусы, индексация, действия администратора базы знаний.
- `Сверка`: сравнение параметров по источникам и статусы `совпадает`, `внимание`, `расхождение`, `на проверку`.
- `История`: запрос, ответ, пользователь, источники, статус, подготовка под поиск и экспорт.
- `QA`: метрики OCR, retrieval, источников, очереди ручной проверки и latency.
- `Видеоинструкция`: полноэкранный demo-walkthrough для показа сценария работы.

Создать `.env.local`:

Если переменная не задана, frontend использует `http://localhost:8000`.

## Что нужно от backend

Backend должен быть локальным или размещенным в контуре заказчика. По ТЗ ожидаемый стек: `FastAPI`, `PostgreSQL`, `pgvector`, `Celery + Redis`, OCR/RAG pipeline.

Общие требования:

- Все ответы API возвращаются в JSON.
- Ошибки возвращаются в едином формате, чтобы UI мог показывать понятный статус.
- Chat/RAG ответ должен укладываться в требование ТЗ до `30 секунд`; для хорошего UX лучше целиться в `6-15 секунд`.
- Каждый RAG-ответ должен содержать источники. Ответ без источников считается неполным.
- Если данных недостаточно, backend возвращает `needs_clarification`, а не пытается угадывать.
- Если источники конфликтуют, backend возвращает `source_conflict`.
- Backend должен отдавать `document`, `section`, `page`, `version`, `text`, `confidence`, чтобы инженер мог проверить ответ.
- Для быстрого и точного поиска нужен hybrid retrieval: keyword search, vector search через `pgvector`, metadata filters, rerank, page anchors.

## API-контракт

### `POST /chat`

Назначение: основной диалог инженера с RAG-системой.

Request:

```json
{
  "q": "Какая минимальная толщина листа для проекта 21900M2?",
  "context": {
    "project": "21900M2",
    "document_type": "norm",
    "version": "actual"
  }
}
```

Response:

```json
{
  "id": "msg-123",
  "role": "assistant",
  "content": "Краткий ответ только по найденным источникам.",
  "timestamp": "10:01",
  "status": "answered",
  "limitation": "Финальное инженерное решение принимает сотрудник.",
  "citations": [
    {
      "id": "c-1",
      "document": "ГОСТ Р 54382-2011",
      "section": "Раздел 4.2.3",
      "page": 12,
      "text": "Фрагмент источника...",
      "version": "2023.1",
      "confidence": 0.91
    }
  ]
}
```

Допустимые статусы ответа:

- `answered`: ответ найден.
- `needs_clarification`: нужно уточнить проект, документ, версию или параметр.
- `insufficient_data`: данных недостаточно.
- `source_conflict`: источники конфликтуют.

### `GET /search`

Назначение: поиск по базе знаний.

Query params:

```text
q=толщина листа
type=PDF
version=actual
source=gdrive_source
limit=20
```

Response:

```json
[
  {
    "id": "doc-1",
    "name": "ГОСТ Р 54382-2011",
    "type": "PDF",
    "version": "2023.1",
    "source": "Нормативная база",
    "page": 12,
    "section": "Раздел 4.2.3",
    "fragment": "Найденный фрагмент...",
    "relevance": 0.92
  }
]
```

### `GET /documents`

Назначение: реестр документов и администрирование базы НСИ.

Response:

```json
[
  {
    "id": "d1",
    "name": "Регламент сборки судна",
    "type": "PDF",
    "version": "v2.0",
    "source": "Нормативная база",
    "ocrStatus": "Завершено",
    "indexStatus": "Индексировано",
    "updatedAt": "2026-04-20"
  }
]
```

Нужные backend actions:

- `POST /documents/upload`
- `POST /documents/{id}/reprocess-ocr`
- `POST /documents/{id}/reindex`
- `GET /documents/{id}/preview?page=12`

### `GET /source-preview`

Назначение: показать страницу или фрагмент первоисточника.

Query params:

```text
document_id=d1
page=12
chunk_id=c1
```

Response:

```json
{
  "document": "ГОСТ Р 54382-2011",
  "page": 12,
  "section": "Раздел 4.2.3",
  "text": "Фрагмент страницы...",
  "bbox": [120, 240, 500, 320],
  "file_url": "/files/d1?page=12"
}
```

Для точного preview backend должен хранить `document_id`, `page`, `chunk_id`, `section`, `bbox` и безопасный internal URL к файлу.

Назначение: история запросов и ответов.

Response:

```json

  {
```

Назначение: обратная связь по ответу.

```json

}
```

## Что важно для скорости и точности

- Нормализовать документы до единой структуры: файл, версия, раздел, страница, таблица, фрагмент.
- Хранить не только текст chunk, но и метаданные: источник, версия, страница, раздел, bbox, дата обновления.
- Разделять индекс по типам документов: нормативные документы, внутренние документы, чертежи, спецификации.
- Делать rerank найденных фрагментов перед генерацией ответа.
- Проверять, что ответ сформирован только по выбранным источникам.
- Для старых сканов использовать OCR с layout detection, а не только простой Tesseract.
- Логировать каждый запрос, найденные фрагменты, итоговый ответ, latency и пользовательский feedback.

## Сценарий демонстрации

1. Инженер открывает `Чат` и задает широкий вопрос.
2. Ассистент просит уточнить проект или документ, если контекста мало.
3. Инженер уточняет контекст.
4. Ассистент возвращает короткий ответ, статус и источники.
5. Инженер открывает источник и проверяет фрагмент.
6. Инженер ставит оценку ответа.
7. Тимлид переходит в `История`, `Реестр`, `QA` и смотрит, что было найдено, какие документы обработаны и где есть риски качества.
